IPC — Shared Memory

Cơ chế IPC nhanh nhất: nhiều process cùng trỏ đến một vùng RAM vật lý — không có kernel copy.

Shared Memory — chia sẻ vùng RAM trực tiếp
Thay vì copy dữ liệu qua kernel, các process cùng nhìn vào một vùng RAM vật lý. Không có trung gian.
Tại sao nó nhanh hơn tất cả IPC còn lại
Với Pipe và Socket: dữ liệu phải đi qua 2 lần copy.
— Process A: user buffer → kernel buffer  (copy 1)
— Process B: kernel buffer → user buffer  (copy 2)

Với Shared Memory: không có copy nào cả. Process A ghi thẳng vào địa chỉ RAM. Process B đọc từ cùng địa chỉ RAM đó. Kernel chỉ thiết lập "bản đồ" một lần ban đầu, sau đó không can thiệp vào từng lần đọc/ghi.

shm_open() và mmap() — từng tham số
Hai hàm này luôn đi cùng nhau. shm_open tạo "file" trong RAM, mmap ánh xạ vào address space.
Luồng sử dụng POSIX Shared Memory
shm_open() — tạo hoặc mở shared memory object. Trả về fd, giống open() với file thật nhưng object nằm trong RAM (tmpfs tại /dev/shm/).
ftruncate() — đặt kích thước. Object mới tạo có size = 0, phải gọi ftruncate() trước khi mmap(). Chỉ cần gọi 1 lần ở phía tạo (producer/server).
mmap() — ánh xạ object vào address space. Sau bước này, con trỏ trả về trỏ trực tiếp vào RAM vật lý.
Đọc/ghi qua con trỏ — nhưng phải có lock nếu có nhiều process đọc/ghi đồng thời.
munmap() + close() + shm_unlink() — dọn dẹp. munmap bỏ ánh xạ, close đóng fd, shm_unlink xóa object khỏi /dev/shm/.
shm_open()
#include <sys/mman.h> #include <fcntl.h> int shm_open(const char *name, int oflag, mode_t mode);
name
Tên object, phải bắt đầu bằng / và không chứa / nào khác. Ví dụ: "/mysensor". Object xuất hiện tại /dev/shm/mysensor — kiểm tra bằng ls /dev/shm/.
oflag
O_CREATTạo object nếu chưa tồn tại. Bỏ flag này ở phía consumer — chỉ mở object đã có.
O_RDWRĐọc và ghi — cần cho ftruncate() và mmap với PROT_WRITE.
O_RDONLYChỉ đọc — dùng ở consumer không cần ghi.
O_EXCLKết hợp O_CREAT: lỗi EEXIST nếu đã tồn tại. Đảm bảo chỉ 1 process tạo.
mode
Permission bits khi O_CREAT. 0600 (chỉ owner), 0660 (owner+group), 0666 (tất cả). Bị mask bởi umask.
Trả về: fd ≥ 0 thành công, -1 lỗi. EEXIST (có O_EXCL), ENOENT (không O_CREAT mà chưa có)
mmap()
#include <sys/mman.h> void *mmap(void *addr, size_t length, int prot, int flags, int fd, off_t offset);
addr
Hầu như luôn truyền NULL — để kernel tự chọn địa chỉ.
length
Số bytes muốn map. Phải ≤ kích thước đã ftruncate(). Kernel tự làm tròn lên page size.
prot
PROT_READCho phép đọc.
PROT_WRITECho phép ghi. Thường kết hợp: PROT_READ | PROT_WRITE.
PROT_NONEKhông có quyền gì — dùng cho guard page.
flags
MAP_SHAREDBắt buộc khi dùng IPC. Ghi hiển thị ngay cho tất cả process khác map cùng object.
MAP_PRIVATECopy-on-write — ghi không ảnh hưởng process khác. Dùng để đọc file không muốn thay đổi.
MAP_ANONYMOUSKhông có file backing — tạo vùng nhớ ẩn danh (fd=-1). Chia sẻ giữa parent/child sau fork().
MAP_LOCKEDLock pages vào RAM, không swap ra đĩa. Quan trọng cho real-time embedded — tránh page fault gây jitter.
MAP_POPULATEPre-fault tất cả pages ngay khi mmap(). Tránh page fault lần đầu — hữu ích cho real-time.
fd
fd từ shm_open(). Dùng -1 với MAP_ANONYMOUS.
offset
Hầu như luôn là 0. Nếu khác 0 phải là bội số của page size (thường 4096).
Trả về: con trỏ đến vùng nhớ, hoặc MAP_FAILED = (void*)-1 khi lỗi
Không so sánh với NULL: mmap() thất bại trả về MAP_FAILED = (void*)-1, không phải NULL. Kiểm tra sai (if (!ptr)) sẽ bỏ qua lỗi và dereference con trỏ invalid — undefined behavior.
munmap() và shm_unlink()
int munmap(void *addr, size_t length); int shm_unlink(const char *name);
munmap
Bỏ ánh xạ khỏi address space của process này. Không xóa object — process khác vẫn dùng được. Phải truyền đúng addr và length từ lúc mmap().
shm_unlink
Xóa tên object khỏi /dev/shm/. Kernel thực sự giải phóng RAM khi tất cả process đã munmap() và close(). Giống unlink() với file thông thường.
Link lúc compile: shm_open, shm_unlink, sem_open... nằm trong librt. Thêm -lrt -lpthread khi compile:
gcc producer.c -o producer -lrt -lpthread
Trên glibc ≥ 2.17 có thể không cần -lrt, nhưng nên thêm cho chắc.
Race condition — lý do bắt buộc phải có lock
Phần quan trọng nhất và dễ sai nhất. Hiểu rõ trước khi viết code.
Tại sao không thể đọc/ghi mà không có lock?
Với Pipe và Socket, kernel serialize tất cả thao tác — chỉ 1 bên ghi tại một thời điểm. Với Shared Memory, kernel không biết bạn đang đọc hay ghi — nó chỉ thấy CPU đang truy cập địa chỉ RAM. Nếu 2 CPU đồng thời đọc và ghi cùng địa chỉ, kết quả là undefined behavior.
Ví dụ race condition cụ thể — counter++
Process A (writer)
Process B (reader)
shm->count = shm->count + 1;
↓ CPU dịch thành 3 lệnh assembly:
MOV eax, [shm->count] ; đọc = 5
ADD eax, 1 ; eax = 6
← Context switch! B đọc count = 5
MOV [shm->count], eax ; ghi 6
B đọc count = 5 — GIÁ TRỊ SAI!
Ngay cả với int cũng không an toàn. Lệnh count++ trông gọn nhưng không atomic — CPU phải read-modify-write. Context switch xảy ra giữa bất kỳ 2 lệnh machine nào.
Ba dạng race condition thường gặp
Read-Modify-Write race (phổ biến nhất): Như ví dụ trên. counter++, enqueue vào ring buffer, cập nhật checksum — tất cả đều cần lock.
Partial write race: Ghi struct lớn không atomic. Reader đọc khi struct đang ghi dở — nhận dữ liệu nửa cũ nửa mới. Ví dụ: struct {float x, y, z} — x mới nhưng y, z cũ.
Reordering race: CPU và compiler có thể sắp xếp lại thứ tự ghi. Writer ghi data trước rồi set flag "ready" — nhưng CPU có thể ghi flag trước data. Cần __sync_synchronize() hoặc atomic_thread_fence().
Giải pháp: 3 cách đồng bộ hoá
POSIX Semaphore — đơn giản nhất. Đặt sem_t trong shared memory, pshared=1. Dùng như mutex: sem_wait trước, sem_post sau.
pthread_mutex với PTHREAD_PROCESS_SHARED — hiệu năng cao hơn semaphore khi contention thấp vì dùng futex (không vào kernel nếu không có contention).
C11 Atomic (_Atomic) — cho dữ liệu nhỏ (int, pointer). CPU đảm bảo atomic ở mức hardware, không cần lock. Zero overhead. Không dùng cho struct lớn.
POSIX Semaphore — đồng bộ Shared Memory
Semaphore là bộ đếm atomic: sem_wait() giảm 1 (block nếu = 0), sem_post() tăng 1.
Hai loại POSIX Semaphore
Unnamed (pshared) — phổ biến nhất
Đặt trực tiếp sem_t trong shared memory. Không có tên.
API: sem_init(pshared=1), sem_destroy()
Dùng khi: semaphore đặt trong SHM — cách phổ biến nhất.
Named semaphore
Có tên /mysem, tồn tại trong hệ thống.
API: sem_open(), sem_close(), sem_unlink()
Dùng khi: process không liên quan cần chia sẻ semaphore mà không dùng SHM.
sem_init() — khởi tạo trong SHM
#include <semaphore.h> int sem_init(sem_t *sem, int pshared, unsigned int value);
sem
Con trỏ đến sem_t nằm trong shared memory. Không dùng sem_t trên stack hay heap thông thường — process khác không thấy.
pshared
0 = chỉ dùng trong cùng process (giữa thread).
1 = chia sẻ giữa các process — bắt buộc khi đặt trong SHM.
value
1 = mutex (chỉ 1 bên vào tại một lúc).
0 = blocking ngay — dùng cho producer/consumer signaling.
N = cho phép N bên vào đồng thời.
Trả về: 0 thành công, -1 lỗi. ENOSYS: kernel không hỗ trợ pshared
sem_wait(), sem_post(), sem_trywait(), sem_timedwait()
int sem_wait(sem_t *sem); /* Giảm 1, block nếu == 0 */ int sem_post(sem_t *sem); /* Tăng 1, wake 1 waiter */ int sem_trywait(sem_t *sem); /* Giảm 1, EAGAIN nếu == 0 (non-block) */ int sem_timedwait(sem_t *sem, const struct timespec *abs_timeout); /* Có timeout tuyệt đối */
sem_wait()
Nếu semaphore > 0: giảm 1 và tiếp tục ngay. Nếu = 0: block cho đến khi sem_post(). Nếu bị signal ngắt: trả về -1 với errno == EINTR — cần retry trong vòng lặp.
sem_post()
Tăng semaphore lên 1. Wake một waiter nếu có. An toàn gọi từ signal handler — async-signal-safe.
sem_timedwait()
Timeout tuyệt đối (không phải relative duration). Dùng clock_gettime(CLOCK_REALTIME, &ts) rồi cộng thêm thời gian. Trả về ETIMEDOUT khi quá hạn.
Trả về: 0 thành công, -1 lỗi. Luôn kiểm tra EINTR với sem_wait()
Pattern đúng: sem_wait() trong vòng lặp xử lý EINTR
/* SAI: gọi 1 lần — có thể bị EINTR ngắt và bỏ qua */
sem_wait(&shm->lock);

/* ĐÚNG: retry khi bị signal ngắt */
while (sem_wait(&shm->lock) == -1) {
    if (errno == EINTR) continue;  /* signal ngắt, thử lại */
    perror("sem_wait");            /* lỗi thực sự */
    exit(1);
}
/* ← critical section: chỉ 1 process vào */
/* ... đọc/ghi shared memory ... */
sem_post(&shm->lock);
/* ← process khác có thể vào */
Pattern producer-consumer với 2 semaphore
Dùng 2 semaphore để đồng bộ không cần polling:
— sem_empty = số slot trống (bắt đầu = N)
— sem_full = số slot có dữ liệu (bắt đầu = 0)

Producer: sem_wait(empty) → ghi → sem_post(full)
Consumer: sem_wait(full) → đọc → sem_post(empty)

Consumer block tại sem_wait(full) khi không có data. Không polling, không waste CPU.
Code đầy đủ, chạy được
Producer ghi sensor data, Consumer đọc và xử lý. Hai process độc lập, không có quan hệ cha-con.
shared.h — include vào cả producer và consumer
#pragma once
#include <semaphore.h>
#include <stdint.h>
#include <time.h>

#define SHM_NAME  "/sensor_shm"
#define SHM_SIZE  sizeof(SharedData)

typedef struct {
    /* ── Đồng bộ ─────────────────────────── */
    sem_t    lock;          /* mutex bảo vệ toàn bộ struct */
    sem_t    data_ready;    /* consumer block ở đây chờ data */

    /* ── Metadata ────────────────────────── */
    uint32_t seq;           /* tăng mỗi lần ghi — consumer detect miss */
    uint8_t  valid;         /* 1 = có dữ liệu hợp lệ */

    /* ── Dữ liệu ─────────────────────────── */
    float    temperature;
    float    pressure;
    float    humidity;
    struct timespec timestamp;
} SharedData;
producer.c — gcc producer.c -o producer -lrt -lpthread
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <unistd.h>
#include <errno.h>
#include <sys/mman.h>
#include <fcntl.h>
#include "shared.h"

int main(void) {
    /* ── Bước 1: Tạo shared memory object ─────────────── */
    int fd = shm_open(SHM_NAME,
                      O_CREAT | O_RDWR,
                      0660);
    if (fd == -1) { perror("shm_open"); exit(1); }

    /* ── Bước 2: Đặt kích thước ────────────────────────── */
    if (ftruncate(fd, (off_t)SHM_SIZE) == -1) {
        perror("ftruncate"); close(fd); exit(1);
    }

    /* ── Bước 3: Map vào address space ─────────────────── */
    SharedData *shm = mmap(NULL, SHM_SIZE,
                           PROT_READ | PROT_WRITE,
                           MAP_SHARED, fd, 0);
    if (shm == MAP_FAILED) {  /* KHÔNG kiểm tra == NULL */
        perror("mmap"); close(fd); exit(1);
    }
    close(fd);  /* fd không cần nữa sau khi mmap */

    /* ── Bước 4: Khởi tạo struct và semaphore ──────────── */
    memset(shm, 0, SHM_SIZE);
    sem_init(&shm->lock,       1, 1);  /* mutex: mở ban đầu */
    sem_init(&shm->data_ready, 1, 0);  /* consumer block ở đây */
    shm->valid = 0;
    printf("[Producer] SHM sẵn sàng: %s\n", SHM_NAME);

    /* ── Bước 5: Vòng lặp ghi dữ liệu ─────────────────── */
    for (uint32_t i = 1; i <= 10; i++) {
        sleep(1);  /* giả lập đọc sensor mỗi 1 giây */

        /* Lấy lock trước khi ghi */
        while (sem_wait(&shm->lock) == -1)
            if (errno != EINTR) { perror("sem_wait"); exit(1); }

        /* ← critical section */
        shm->seq         = i;
        shm->temperature = 20.0f + (float)(i % 10);
        shm->pressure    = 1013.25f + (float)(i % 5);
        shm->humidity    = 60.0f + (float)(i % 20);
        clock_gettime(CLOCK_MONOTONIC, &shm->timestamp);
        shm->valid       = 1;
        /* ← kết thúc critical section */

        printf("[Producer] Ghi seq=%u, temp=%.1f°C\n",
               i, shm->temperature);

        sem_post(&shm->lock);       /* nhả lock */
        sem_post(&shm->data_ready); /* báo consumer có data */
    }

    /* ── Cleanup ────────────────────────────────────────── */
    /* Chờ consumer đọc xong trước khi xóa */
    sleep(1);
    sem_destroy(&shm->data_ready);
    sem_destroy(&shm->lock);
    munmap(shm, SHM_SIZE);
    shm_unlink(SHM_NAME);  /* xóa khỏi /dev/shm/ */
    printf("[Producer] Done.\n");
    return 0;
}
consumer.c — gcc consumer.c -o consumer -lrt -lpthread
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <errno.h>
#include <sys/mman.h>
#include <fcntl.h>
#include "shared.h"

int main(void) {
    /* Consumer chỉ mở — không tạo (không có O_CREAT) */
    int fd = shm_open(SHM_NAME, O_RDWR, 0);
    if (fd == -1) {
        perror("shm_open — producer chưa chạy?");
        exit(1);
    }

    SharedData *shm = mmap(NULL, SHM_SIZE,
                           PROT_READ | PROT_WRITE,
                           MAP_SHARED, fd, 0);
    if (shm == MAP_FAILED) { perror("mmap"); exit(1); }
    close(fd);

    printf("[Consumer] Đã attach SHM, đang chờ data...\n");

    while (1) {
        /* Block ở đây cho đến khi producer sem_post */
        while (sem_wait(&shm->data_ready) == -1)
            if (errno != EINTR) { perror("sem_wait"); exit(1); }

        /* Lấy lock để đọc an toàn */
        while (sem_wait(&shm->lock) == -1)
            if (errno != EINTR) { perror("sem_wait"); exit(1); }

        /* ← critical section */
        uint32_t seq = shm->seq;
        float temp   = shm->temperature;
        float pres   = shm->pressure;
        float hum    = shm->humidity;
        /* ← kết thúc critical section */

        sem_post(&shm->lock);

        printf("[Consumer] seq=%u | %.1f°C | %.2fhPa | %.1f%%\n",
               seq, temp, pres, hum);

        if (seq >= 10) break;
    }

    munmap(shm, SHM_SIZE);
    printf("[Consumer] Done.\n");
    return 0;
}

Shared Memory trong Embedded Linux
Ứng dụng thực tế, ràng buộc phần cứng, và những điểm phải chú ý.
Use case điển hình
Camera frame buffer
Camera driver ghi frame (1920×1080×3 ≈ 6MB) vào SHM. ISP process xử lý. Display process render. Không thể dùng Pipe — copy 6MB×30fps = 180MB/s chỉ để copy.

Pattern: Double buffer — ghi slot B khi reader đang đọc slot A.
Sensor aggregator
Nhiều sensor process ghi vào struct trong SHM. Dashboard process đọc và hiển thị. Yêu cầu latency thấp — không thể chờ socket roundtrip.

Pattern: Struct với seq number và lock. Consumer kiểm tra seq để detect missed update.
Ràng buộc quan trọng trên embedded
/dev/shm cần tmpfs: POSIX SHM lưu vào /dev/shm. Trên Buildroot mặc định có thể không mount tmpfs tại đây. Kiểm tra: mount | grep /dev/shm. Nếu không có, thêm vào /etc/fstab:
tmpfs /dev/shm tmpfs defaults,size=64M 0 0
Hoặc dùng mmap() + open() với file thường trong /tmp thay shm_open().
MAP_LOCKED cho real-time: Trên hệ thống real-time (robotics, motor control), page fault trong critical section gây jitter hàng ms — thảm họa. Dùng MAP_LOCKED | MAP_POPULATE để lock pages vào RAM ngay từ đầu. Cần mlockall(MCL_CURRENT | MCL_FUTURE) ở đầu chương trình và quyền CAP_IPC_LOCK.
RAM giới hạn: /dev/shm mặc định chiếm 50% RAM. Trên board 256MB, mỗi frame buffer 6MB chiếm đáng kể. Đặt giới hạn size tmpfs rõ ràng trong fstab. Kiểm tra: df -h /dev/shm.
Pattern: mmap() với file thường khi không có /dev/shm
Thay thế shm_open bằng open() + mmap() — portable hơn trên Buildroot
/* Thay vì shm_open("/mysensor", ...) */
int fd = open("/tmp/sensor_data.bin",
              O_CREAT | O_RDWR, 0660);
ftruncate(fd, sizeof(SharedData));

SharedData *shm = mmap(NULL, sizeof(SharedData),
                        PROT_READ | PROT_WRITE,
                        MAP_SHARED, fd, 0);
/* Hoạt động y hệt POSIX SHM nhưng:
   + Không cần /dev/shm tmpfs
   + Có thể debug: xxd /tmp/sensor_data.bin
   - File tồn tại sau reboot (có thể là feature hoặc bug)
   - Có thể swap ra eMMC nếu không MAP_LOCKED
   Giải pháp: đặt file trong /tmp (nếu /tmp là tmpfs) */
Cleanup đúng cách trong signal handler
Tránh SHM bị orphan khi daemon crash
static SharedData *g_shm  = NULL;
static size_t      g_size = 0;

void cleanup_handler(int sig) {
    if (g_shm) {
        sem_destroy(&g_shm->data_ready);
        sem_destroy(&g_shm->lock);
        munmap(g_shm, g_size);
        g_shm = NULL;
    }
    shm_unlink(SHM_NAME);
    _exit(sig == SIGTERM ? 0 : 1);  /* _exit() là async-signal-safe */
}

int main(void) {
    struct sigaction sa = { .sa_handler = cleanup_handler };
    sigaction(SIGTERM, &sa, NULL);
    sigaction(SIGINT,  &sa, NULL);
    /* ... */
}

/* Nếu producer crash mà không cleanup:
   - SHM object còn trong /dev/shm — ls /dev/shm/ để thấy
   - Consumer mở lại được nhưng seq/valid có thể sai
   - Script khởi động: rm -f /dev/shm/sensor_shm trước launch */
Bước tiếp theo: Sau Shared Memory, IPC cuối cần học là POSIX Message Queue — cho pattern publish-subscribe và event-driven architecture. Message Queue giải quyết bài toán SHM không giải được: nhiều consumer độc lập nhận message khác nhau, có priority ordering, không cần polling.
