Unix Domain Socket — socket không qua mạng
Giao tiếp bidirectional giữa các process trên cùng một máy, qua file path thay vì IP:port.
So sánh trực quan với Pipe
Pipe là ống nước một chiều: chỉ chảy từ A sang B, không thể gửi ngược lại trên cùng ống.

Unix Socket là điện thoại: sau khi kết nối, cả hai bên đều có thể nói và nghe bất cứ lúc nào. Một server có thể nghe nhiều client gọi đến cùng lúc — mỗi cuộc gọi là một kênh riêng biệt.

Ba loại socket type
SOCK_STREAM
Giống TCP: byte stream liên tục, không có message boundary. Gửi 100 bytes rồi 50 bytes, receiver có thể đọc 150 bytes một lần hoặc 30+120 bytes.

Đảm bảo thứ tự, không mất dữ liệu.
Dùng khi: gửi dữ liệu lớn liên tục, stream protocol có header.
SOCK_DGRAM
Giống UDP: mỗi sendto() là 1 message riêng biệt — receiver nhận đúng 1 message mỗi lần recvfrom(). Giữ message boundary.

Dùng khi: gửi lệnh ngắn, event notification, struct cố định, không cần kết nối trước.
SOCK_SEQPACKET — kết hợp ưu điểm cả hai
SOCK_SEQPACKET = connection-oriented + message boundary. Giống STREAM ở chỗ cần connect/accept, nhưng giống DGRAM ở chỗ giữ nguyên ranh giới message.

Ví dụ: gửi 100 bytes rồi 50 bytes → receiver luôn nhận đúng 2 lần: 100 bytes và 50 bytes, không bao giờ bị gộp hay tách.

Ưu điểm so với STREAM: không cần tự thiết kế framing protocol (thêm header length), kernel đã giữ boundary cho bạn.
Ưu điểm so với DGRAM: có kết nối rõ ràng, flow control, đảm bảo thứ tự và không mất dữ liệu.

Dùng khi: giao tiếp IPC dạng command-response, gửi struct cố định qua connection, cần cả reliability lẫn message boundary.
Chọn type nào? Nếu cần message rõ ràng (struct, JSON command) + có kết nối → SOCK_SEQPACKET là lựa chọn lý tưởng nhất. Nếu chỉ fire-and-forget → SOCK_DGRAM. Nếu stream data lớn (file, video) → SOCK_STREAM.
Vòng đời kết nối SOCK_STREAM
Mỗi bước có lý do kỹ thuật rõ ràng — hiểu tại sao thì nhớ API mãi.
Tại sao cần đủ 4 bước phía server?
socket() — tạo endpoint trống trong kernel. Chưa có địa chỉ, chưa nghe gì cả. Trả về 1 file descriptor.
bind() — gắn địa chỉ (path trên filesystem) vào endpoint. Tạo ra file app.sock trên disk. Nếu file đã tồn tại → lỗi EADDRINUSE.
listen() — bật chế độ "đang nhận khách". Kernel tạo một accept queue với sức chứa tối đa backlog client đang chờ. Client connect trước khi server accept sẽ nằm trong queue này.
accept() — lấy 1 client ra khỏi accept queue. Block cho đến khi có client. Trả về 1 fd mới riêng cho client đó — srv_fd vẫn tiếp tục nghe client mới. Đây là điểm khác biệt then chốt so với Pipe.
accept queue & backlog: listen(fd, backlog) — backlog là số kết nối kernel đã chấp nhận nhưng server chưa kịp accept(). Nếu queue đầy, client mới nhận ECONNREFUSED. Với embedded single-client: backlog = 1 là đủ.
API chi tiết — từng tham số
Tất cả API cần biết: socket, bind, listen, accept, connect, send, recv, sendto, recvfrom, sendmsg, recvmsg, close, shutdown, unlink, getsockopt, setsockopt, fcntl, poll, epoll, signalfd, socketpair.
socket() — tạo endpoint
#include <sys/socket.h> int socket(int domain, int type, int protocol);
domain
AF_UNIX (hay AF_LOCAL — cùng giá trị) = Unix Domain Socket, chỉ giao tiếp trên cùng máy, không qua network stack. Khác với AF_INET (IPv4) hay AF_INET6 (IPv6).
type
SOCK_STREAM — byte stream, có kết nối, đảm bảo thứ tự (như TCP)
SOCK_DGRAM — datagram, không kết nối, giữ message boundary
SOCK_SEQPACKET — có kết nối + giữ message boundary

OR thêm flags:
SOCK_NONBLOCK — non-blocking ngay khi tạo
SOCK_CLOEXEC — tự đóng khi exec() — luôn dùng
protocol
Luôn là 0 với Unix socket.
Trả về: fd ≥ 0 thành công, -1 lỗi. EMFILE: hết fd, ENFILE: system limit
bind() — gắn địa chỉ
#include <sys/socket.h> #include <sys/un.h> int bind(int sockfd, const struct sockaddr *addr, socklen_t addrlen);
addr
Con trỏ đến struct sockaddr_un — cast sang (struct sockaddr *).
sun_family: luôn là AF_UNIX
sun_path: chuỗi path — tối đa 108 ký tự trên Linux
addrlen
Dùng sizeof(struct sockaddr_un) là đủ và an toàn.
Trả về: 0 thành công, -1 lỗi. EADDRINUSE: socket file đã tồn tại → cần unlink() trước
Khởi tạo struct sockaddr_un đúng chuẩn
#include <sys/un.h>
#include <string.h>

#define SOCK_PATH "/tmp/myapp.sock"

struct sockaddr_un addr;
memset(&addr, 0, sizeof(addr));        /* Xóa toàn bộ struct trước */
addr.sun_family = AF_UNIX;
strncpy(addr.sun_path, SOCK_PATH,
        sizeof(addr.sun_path) - 1);    /* Không tràn 108 ký tự */

unlink(SOCK_PATH);                     /* Xóa socket file cũ nếu còn */

if (bind(srv_fd, (struct sockaddr *)&addr, sizeof(addr)) == -1) {
    perror("bind");
    close(srv_fd);
    return -1;
}
Tại sao bắt buộc memset(&addr, 0, ...) trước khi dùng?
struct sockaddr_un có trường sun_path[108]. Nếu không memset, các byte sau chuỗi path thực tế sẽ chứa rác từ stack. Kernel so sánh toàn bộ sun_path khi matching socket → nếu có rác, connect() từ client sẽ không khớp với bind() của server dù path nhìn giống nhau. Đây là bug rất khó debug vì mọi thứ trông đúng nhưng connect() trả về ECONNREFUSED.
listen() và accept4()
int listen(int sockfd, int backlog); int accept4(int sockfd, struct sockaddr *addr, socklen_t *addrlen, int flags);
backlog
Số kết nối tối đa trong accept queue chờ server gọi accept(). Bị cap bởi /proc/sys/net/core/somaxconn (mặc định 128). Embedded single-client: backlog = 1.
flags (accept4)
SOCK_CLOEXEC | SOCK_NONBLOCK — dùng accept4() thay accept() để set flags atomic, tránh race condition trong multi-thread.
accept4 trả về: fd mới ≥ 0, -1 lỗi. EAGAIN: non-blocking và chưa có client
send() và recv() — flags quan trọng
ssize_t send(int sockfd, const void *buf, size_t len, int flags); ssize_t recv(int sockfd, void *buf, size_t len, int flags);
flags (send)
0 — mặc định, giống write()
MSG_NOSIGNAL — không gửi SIGPIPE khi peer đóng. Luôn dùng trong daemon.
MSG_DONTWAIT — non-blocking cho lần này (không đổi trạng thái socket)
flags (recv)
0 — mặc định
MSG_WAITALL — block cho đến khi đủ len bytes. Hữu ích khi biết trước kích thước.
MSG_PEEK — xem dữ liệu mà không xóa khỏi buffer
Trả về: bytes đọc/ghi, 0 = peer đóng (EOF), -1 lỗi
MSG_NOSIGNAL là bắt buộc với daemon: Khi client đột ngột thoát, lần send() kế tiếp từ server sẽ nhận SIGPIPE — mặc định kill process. Dùng MSG_NOSIGNAL để send() trả về -1 với errno=EPIPE thay vì crash.
Xử lý partial send/recv — lỗi phổ biến nhất
Với SOCK_STREAM, send() có thể chỉ gửi một phần dữ liệu, đặc biệt khi buffer kernel đầy. Ví dụ: gửi 1000 bytes nhưng send() chỉ trả về 600. Nếu không kiểm tra, 400 bytes còn lại bị mất — đây là bug rất phổ biến.

Tương tự, recv() có thể trả về ít hơn len bytes ngay cả khi dữ liệu đang đến. Cần vòng lặp để đảm bảo nhận đủ.
send_all() — đảm bảo gửi hết data, dùng thay send() trực tiếp
/* Gửi đúng 'len' bytes. Trả về 0 nếu thành công, -1 nếu lỗi */
int send_all(int fd, const void *buf, size_t len) {
    const char *p = (const char *)buf;
    size_t remaining = len;

    while (remaining > 0) {
        ssize_t n = send(fd, p, remaining, MSG_NOSIGNAL);
        if (n == -1) {
            if (errno == EINTR) continue;   /* bị signal ngắt, thử lại */
            return -1;                       /* lỗi thật: EPIPE, ECONNRESET... */
        }
        p += n;
        remaining -= (size_t)n;
    }
    return 0;
}

/* Nhận đúng 'len' bytes. Trả về 0 nếu đủ, -1 lỗi, 1 nếu peer đóng sớm */
int recv_all(int fd, void *buf, size_t len) {
    char *p = (char *)buf;
    size_t remaining = len;

    while (remaining > 0) {
        ssize_t n = recv(fd, p, remaining, 0);
        if (n == -1) {
            if (errno == EINTR) continue;
            return -1;
        }
        if (n == 0) return 1;  /* peer đóng kết nối trước khi gửi đủ */
        p += n;
        remaining -= (size_t)n;
    }
    return 0;
}
Khi nào KHÔNG cần send_all? Với SOCK_DGRAM và SOCK_SEQPACKET, kernel đảm bảo gửi toàn bộ hoặc lỗi — không có partial send. Chỉ SOCK_STREAM mới cần pattern này.

connect() — kết nối đến server (phía client)
#include <sys/socket.h> #include <sys/un.h> int connect(int sockfd, const struct sockaddr *addr, socklen_t addrlen);
sockfd
fd trả về từ socket(). Phải cùng domain (AF_UNIX) và type (SOCK_STREAM hoặc SOCK_SEQPACKET) với server.
addr
Con trỏ đến struct sockaddr_un với sun_path trỏ đến cùng path mà server đã bind(). Phải memset trước khi dùng.
addrlen
sizeof(struct sockaddr_un). Với abstract socket: offsetof(struct sockaddr_un, sun_path) + 1 + strlen(name).
Trả về: 0 thành công, -1 lỗi. ENOENT: socket file không tồn tại. ECONNREFUSED: không ai listen. EACCES: không đủ quyền
Blocking vs Non-blocking connect(): Mặc định, connect() block cho đến khi server accept. Với SOCK_NONBLOCK, connect() return ngay -1 với errno = EINPROGRESS. Dùng poll() hoặc epoll theo dõi POLLOUT để biết khi nào kết nối hoàn tất.
close() và shutdown() — đóng kết nối
#include <unistd.h> int close(int fd); #include <sys/socket.h> int shutdown(int sockfd, int how);
close(fd)
Đóng file descriptor. Nếu là fd cuối cùng tham chiếu socket này → kernel giải phóng tài nguyên. Peer nhận recv() = 0 (EOF).

Chú ý: close() đóng cả 2 chiều (đọc + ghi) cùng lúc. Nếu socket buffer còn data chưa gửi, kernel cố gắng gửi hết trước khi đóng (SO_LINGER kiểm soát hành vi này).
how (shutdown)
SHUT_RD (0) — đóng chiều đọc. recv() sau đó return 0
SHUT_WR (1) — đóng chiều ghi. Peer nhận EOF. Dùng khi đã gửi xong nhưng chưa muốn đóng fd.
SHUT_RDWR (2) — đóng cả hai chiều, giống close() nhưng không giải phóng fd — fd vẫn tồn tại, cần close() riêng
Trả về: 0 thành công, -1 lỗi
close() vs shutdown(): close() đóng fd luôn — không thể dùng lại. shutdown(SHUT_WR) chỉ đóng chiều ghi nhưng vẫn có thể recv() data từ peer. Pattern quan trọng: client gửi xong → shutdown(fd, SHUT_WR) → server nhận EOF và biết client gửi hết → server reply → client recv() reply → close().
unlink() — xóa socket file
#include <unistd.h> int unlink(const char *pathname);
pathname
Path đến socket file cần xóa. Bắt buộc gọi trước bind() nếu file có thể tồn tại từ lần chạy trước. Nếu không xóa → bind() lỗi EADDRINUSE.

Cũng bắt buộc gọi khi server thoát (trong cleanup hoặc signal handler) để dọn dẹp.
Trả về: 0 thành công, -1 lỗi. ENOENT: file không tồn tại — bình thường, có thể ignore
Abstract socket không cần unlink(): Nếu sun_path[0] = '\0' (abstract namespace), không có file thật trên FS → không cần cleanup. Kernel tự dọn khi tất cả fd đóng.
sendto() và recvfrom() — cho SOCK_DGRAM
ssize_t sendto(int sockfd, const void *buf, size_t len, int flags, const struct sockaddr *dest_addr, socklen_t addrlen); ssize_t recvfrom(int sockfd, void *buf, size_t len, int flags, struct sockaddr *src_addr, socklen_t *addrlen);
dest_addr (sendto)
Địa chỉ đích — con trỏ đến struct sockaddr_un của receiver. Với DGRAM, mỗi sendto() phải chỉ rõ gửi đến đâu vì không có kết nối trước.
src_addr (recvfrom)
Kernel điền địa chỉ sender vào đây. Có thể NULL nếu không cần biết ai gửi. Nếu muốn reply, cần lưu địa chỉ này để sendto() lại.
addrlen (recvfrom)
Con trỏ — khởi tạo = sizeof(struct sockaddr_un). Kernel ghi lại kích thước thật của địa chỉ sender.
flags
Giống send()/recv(): 0 mặc định, MSG_DONTWAIT non-blocking, MSG_NOSIGNAL tránh SIGPIPE.
Trả về: bytes gửi/nhận. DGRAM đảm bảo atomic: gửi toàn bộ hoặc lỗi, không partial. -1 lỗi. ENOBUFS: receiver buffer đầy
Client DGRAM muốn nhận reply phải bind(): Nếu client không bind() một path, server không biết gửi reply về đâu (src_addr trong recvfrom() sẽ rỗng). Client cần bind một path tạm: /tmp/client_XXXX.sock rồi unlink() khi kết thúc.
sendmsg() và recvmsg() — ancillary data, SCM_RIGHTS
ssize_t sendmsg(int sockfd, const struct msghdr *msg, int flags); ssize_t recvmsg(int sockfd, struct msghdr *msg, int flags);
struct msghdr
msg_iov / msg_iovlen — data chính (scatter/gather I/O). Phải có ít nhất 1 byte data thật.
msg_control — buffer cho ancillary data (control message)
msg_controllen — kích thước buffer
msg_name / msg_namelen — địa chỉ đích (cho DGRAM, NULL cho STREAM)
struct cmsghdr
Header của mỗi control message trong ancillary data:
cmsg_level = SOL_SOCKET
cmsg_type = SCM_RIGHTS (truyền fd) hoặc SCM_CREDENTIALS (truyền credentials)
cmsg_len = CMSG_LEN(sizeof(int)) cho 1 fd

Dùng macro: CMSG_FIRSTHDR(), CMSG_DATA(), CMSG_SPACE(), CMSG_LEN()
Trả về: bytes data (không tính ancillary), -1 lỗi. Ancillary data chỉ đi kèm — không tính vào return value
Khi nào dùng sendmsg/recvmsg:
— Truyền file descriptor giữa process (SCM_RIGHTS) → xem code chi tiết ở tab "7. Nâng cao"
— Truyền credentials (SCM_CREDENTIALS) cho DGRAM
— Scatter/gather I/O: gửi data từ nhiều buffer rời rạc trong 1 lần system call
Nếu không cần ancillary data, dùng send()/recv() đơn giản hơn.
getsockopt() / setsockopt() — tùy chỉnh socket
#include <sys/socket.h> int getsockopt(int sockfd, int level, int optname, void *optval, socklen_t *optlen); int setsockopt(int sockfd, int level, int optname, const void *optval, socklen_t optlen);
level
Luôn là SOL_SOCKET cho các option dưới đây.
SO_PEERCRED
Chỉ getsockopt. Lấy PID/UID/GID của peer (client) sau accept(). optval = struct ucred *. Kernel cung cấp — client không thể giả mạo. Linux-only. Chỉ với SOCK_STREAM/SEQPACKET.
SO_RCVBUF
Kích thước receive buffer (bytes). optval = int *. Kernel nhân đôi giá trị set. Mặc định: cat /proc/sys/net/core/rmem_default. Quan trọng với DGRAM: buffer đầy → message bị drop.
SO_SNDBUF
Kích thước send buffer (bytes). Ảnh hưởng lượng data có thể ghi trước khi send() block (hoặc EAGAIN nếu non-blocking).
SO_PASSCRED
Bật nhận credentials qua ancillary data (SCM_CREDENTIALS) trong recvmsg(). Cần thiết cho SOCK_DGRAM vì không có SO_PEERCRED. optval = int * (1 = bật).
SO_LINGER
Kiểm soát hành vi close() khi buffer còn data: struct linger { int l_onoff; int l_linger; }. Nếu bật: close() block tối đa l_linger giây chờ gửi hết. Nếu hết thời gian → drop data.
SO_REUSEADDR
Không có tác dụng với AF_UNIX. Đây là lỗi phổ biến khi copy code từ TCP socket. Với Unix socket, luôn dùng unlink() thay vì SO_REUSEADDR.
Trả về: 0 thành công, -1 lỗi
fcntl() — thay đổi thuộc tính fd
#include <fcntl.h> int fcntl(int fd, int cmd, ... /* arg */);
F_GETFL / F_SETFL
Đọc/ghi file status flags. Flags quan trọng cho socket:
O_NONBLOCK — bật chế độ non-blocking
O_ASYNC — bật signal-driven I/O (SIGIO khi data đến)

Pattern chuẩn: int fl = fcntl(fd, F_GETFL, 0); fcntl(fd, F_SETFL, fl | O_NONBLOCK);
F_SETOWN
Đặt process nhận SIGIO/SIGURG. fcntl(fd, F_SETOWN, getpid()). Chỉ cần nếu dùng O_ASYNC (signal-driven I/O).
F_SETFD
Set fd flags — thường để set FD_CLOEXEC. Nhưng nên dùng SOCK_CLOEXEC khi tạo socket thay vì fcntl() riêng — atomic, tránh race condition.
Trả về: phụ thuộc cmd. F_GETFL trả flags, F_SETFL trả 0/-1
poll() — I/O multiplexing cơ bản
#include <poll.h> int poll(struct pollfd *fds, nfds_t nfds, int timeout);
struct pollfd
int fd — file descriptor cần theo dõi. Đặt -1 để bỏ qua entry
short events — sự kiện quan tâm (input):
  POLLIN — có data đọc được hoặc accept() ready
  POLLOUT — có thể ghi mà không block
short revents — sự kiện thực tế xảy ra (output, kernel điền):
  POLLIN, POLLOUT — như trên
  POLLHUP — peer đóng kết nối (luôn được report, không cần đăng ký)
  POLLERR — lỗi trên fd (luôn được report)
nfds
Số phần tử trong mảng fds. Kiểu nfds_t (unsigned long).
timeout
-1 = block vô hạn. 0 = return ngay (poll mode). >0 = timeout milliseconds.
Trả về: số fd có sự kiện (>0), 0 timeout, -1 lỗi. EINTR: bị signal ngắt — nên retry
epoll — I/O multiplexing hiệu năng cao (Linux-only)
#include <sys/epoll.h> int epoll_create1(int flags); int epoll_ctl(int epfd, int op, int fd, struct epoll_event *event); int epoll_wait(int epfd, struct epoll_event *events, int maxevents, int timeout);
epoll_create1
Tạo epoll instance. flags: EPOLL_CLOEXEC (luôn dùng) hoặc 0. Trả về epoll fd — dùng cho epoll_ctl() và epoll_wait(). Cần close() khi kết thúc.
epoll_ctl — op
EPOLL_CTL_ADD — đăng ký fd mới
EPOLL_CTL_MOD — thay đổi events quan tâm
EPOLL_CTL_DEL — gỡ fd (event có thể NULL)
struct epoll_event
uint32_t events — bitmask sự kiện:
  EPOLLIN — có data đọc / accept ready
  EPOLLOUT — có thể ghi
  EPOLLET — Edge-Triggered: chỉ thông báo khi trạng thái thay đổi, không lặp lại. Phải đọc hết data trong 1 lần (loop recv đến EAGAIN). Hiệu quả hơn nhưng khó dùng đúng.
  EPOLLONESHOT — chỉ trigger 1 lần, cần EPOLL_CTL_MOD để re-arm
  EPOLLRDHUP — peer đóng chiều ghi (half-close). Cần Linux ≥ 2.6.17

epoll_data_t data — union chứa user data. Thường dùng .fd để biết fd nào có event, hoặc .ptr trỏ đến struct context tự định nghĩa.
epoll_wait
events: mảng output, kernel điền vào. maxevents: kích thước mảng. timeout: -1 block, 0 non-block, >0 ms.
epoll_wait trả về: số event (>0), 0 timeout, -1 lỗi. Chỉ trả về fd thực sự có sự kiện — O(1) per event
Level-Triggered vs Edge-Triggered:
LT (mặc định): epoll_wait() thông báo liên tục khi fd có data chưa đọc. An toàn, dễ dùng. Nếu recv() chưa đọc hết, lần epoll_wait() sau vẫn thông báo.
ET (EPOLLET): chỉ thông báo 1 lần khi trạng thái đổi (không data → có data). Phải đọc hết (loop đến EAGAIN), nếu không data còn lại sẽ "mất". Hiệu quả hơn nhưng bug nếu quên đọc hết.
Khuyên dùng: LT cho hầu hết trường hợp. ET chỉ khi cần tối ưu hiệu năng tối đa.

unix_server.c — gcc unix_server.c -o server
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <unistd.h>
#include <errno.h>
#include <sys/socket.h>
#include <sys/un.h>

#define SOCK_PATH "/tmp/echo.sock"
#define BUF_SIZE  256

int main(void) {
    /* ── Bước 1: Tạo socket ─────────────────────────── */
    int srv_fd = socket(AF_UNIX,
                        SOCK_STREAM | SOCK_CLOEXEC, 0);
    /* SOCK_CLOEXEC: tự đóng nếu process gọi exec()
       Không cần fcntl() riêng — atomic và an toàn hơn */
    if (srv_fd == -1) { perror("socket"); exit(1); }

    /* ── Bước 2: Bind ───────────────────────────────── */
    struct sockaddr_un addr;
    memset(&addr, 0, sizeof(addr));
    addr.sun_family = AF_UNIX;
    strncpy(addr.sun_path, SOCK_PATH, sizeof(addr.sun_path)-1);

    unlink(SOCK_PATH);   /* Xóa socket file cũ nếu còn từ lần trước */

    if (bind(srv_fd, (struct sockaddr *)&addr, sizeof(addr)) == -1) {
        perror("bind"); close(srv_fd); exit(1);
    }

    /* ── Bước 3: Listen ─────────────────────────────── */
    if (listen(srv_fd, 5) == -1) {
        perror("listen"); close(srv_fd); exit(1);
    }
    printf("Server dang lang nghe tai %s\n", SOCK_PATH);

    /* ── Vòng lặp: nhận từng client ─────────────────── */
    while (1) {
        int cli_fd = accept4(srv_fd, NULL, NULL,
                             SOCK_CLOEXEC);
        /* accept4 với SOCK_CLOEXEC: fd mới cũng tự đóng khi exec
           Dùng accept4() thay accept() để set flag atomic */
        if (cli_fd == -1) { perror("accept4"); continue; }

        printf("Client ket noi, fd=%d\n", cli_fd);

        char buf[BUF_SIZE];
        ssize_t n = recv(cli_fd, buf, sizeof(buf)-1, 0);

        if (n > 0) {
            buf[n] = '\0';
            printf("Nhan: \"%s\"\n", buf);

            /* Echo lại — MSG_NOSIGNAL tránh SIGPIPE */
            if (send(cli_fd, buf, n, MSG_NOSIGNAL) == -1) {
                if (errno == EPIPE)
                    printf("Client dong truoc khi nhan\n");
                else
                    perror("send");
            }
        } else if (n == 0) {
            printf("Client dong ket noi (EOF)\n");
        } else {
            perror("recv");
        }

        close(cli_fd);   /* Đóng kết nối client này */
        /* srv_fd vẫn mở — vòng lặp tiếp tục accept */
    }

    close(srv_fd);
    unlink(SOCK_PATH);
    return 0;
}

#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <unistd.h>
#include <sys/socket.h>
#include <sys/un.h>

#define SOCK_PATH "/tmp/echo.sock"

int main(int argc, char *argv[]) {
    const char *msg = (argc > 1) ? argv[1] : "Xin chao server!";

    /* ── Tạo socket ──────────────────────────────────── */
    int fd = socket(AF_UNIX, SOCK_STREAM | SOCK_CLOEXEC, 0);
    if (fd == -1) { perror("socket"); exit(1); }

    /* ── Connect ─────────────────────────────────────── */
    struct sockaddr_un addr;
    memset(&addr, 0, sizeof(addr));
    addr.sun_family = AF_UNIX;
    strncpy(addr.sun_path, SOCK_PATH, sizeof(addr.sun_path)-1);

    if (connect(fd, (struct sockaddr *)&addr, sizeof(addr)) == -1) {
        /* ENOENT       = server chưa chạy (socket file không tồn tại)
           ECONNREFUSED = server đang chạy nhưng không accept */
        perror("connect"); close(fd); exit(1);
    }
    printf("Da ket noi den %s\n", SOCK_PATH);

    /* ── Gửi ─────────────────────────────────────────── */
    ssize_t sent = send(fd, msg, strlen(msg), MSG_NOSIGNAL);
    if (sent == -1) { perror("send"); close(fd); exit(1); }
    printf("Da gui: \"%s\" (%zd bytes)\n", msg, sent);

    /* ── Nhận echo ───────────────────────────────────── */
    char buf[256];
    ssize_t n = recv(fd, buf, sizeof(buf)-1, 0);
    if (n > 0) {
        buf[n] = '\0';
        printf("Echo tu server: \"%s\"\n", buf);
    }

    close(fd);
    return 0;
}

SOCK_DGRAM — message-based, không cần kết nối
Không có accept(), mỗi sendto() là 1 message riêng biệt, giữ nguyên boundary.
Khác biệt cơ bản so với SOCK_STREAM
Với SOCK_STREAM phải trải qua connect → accept → send/recv → close. Phức tạp nhưng có kết nối rõ ràng.

Với SOCK_DGRAM: không cần kết nối. Client chỉ cần biết path của server rồi sendto() thẳng. Mỗi sendto() là 1 message trọn vẹn — không bao giờ bị tách hoặc gộp với message khác.

Ưu điểm lớn: message boundary được giữ nguyên. Gửi struct 32 bytes → nhận đúng 32 bytes, không cần framing protocol.
Hạn chế của DGRAM
— Có giới hạn kích thước message (~200 KB)
— Nếu receiver không đọc kịp, message có thể bị drop
— Không có flow control tự động
— Để server reply, client cũng phải bind() một path
Khi nào dùng DGRAM
— Gửi struct nhỏ, event notification
— Nhiều client → 1 server, không cần reply
— Logging daemon (nhiều process ghi log vào 1 socket)
— Embedded: sensor → controller (fire-and-forget)
dgram_server.c — nhận struct từ nhiều client, không cần kết nối
#include <sys/socket.h>
#include <sys/un.h>
#include <unistd.h>
#include <stdio.h>
#include <string.h>

#define SERVER_PATH "/tmp/sensor.sock"

typedef struct {
    int   sensor_id;
    float value;
    long  timestamp;
} SensorData;

int main(void) {
    int fd = socket(AF_UNIX, SOCK_DGRAM | SOCK_CLOEXEC, 0);

    struct sockaddr_un addr;
    memset(&addr, 0, sizeof(addr));
    addr.sun_family = AF_UNIX;
    strncpy(addr.sun_path, SERVER_PATH, sizeof(addr.sun_path)-1);

    unlink(SERVER_PATH);
    bind(fd, (struct sockaddr *)&addr, sizeof(addr));
    /* Không cần listen() — DGRAM không có kết nối */

    printf("Sensor server dang lang nghe...\n");

    SensorData data;
    struct sockaddr_un client_addr;
    socklen_t client_len = sizeof(client_addr);

    while (1) {
        /* recvfrom() nhận đúng 1 message, biết địa chỉ client */
        ssize_t n = recvfrom(fd, &data, sizeof(data), 0,
                             (struct sockaddr *)&client_addr,
                             &client_len);
        if (n == (ssize_t)sizeof(data)) {
            printf("Sensor %d: %.2f (t=%ld)\n",
                   data.sensor_id, data.value, data.timestamp);
        }
    }
    close(fd);
    unlink(SERVER_PATH);
    return 0;
}

Pattern thực tế, ràng buộc và ưu tiên khi dùng trên thiết bị nhúng.
Khi nào chuyển từ Pipe sang Unix Socket
Pipe đủ khi giao tiếp đơn giản 1:1 giữa parent/child. Chuyển sang Unix Socket khi:

— Cần nhiều client kết nối đến 1 daemon (nhiều process gửi lệnh đến manager)
— Cần reply từ server về client (bidirectional)
— Hai process không có quan hệ cha-con và cần giao tiếp lâu dài
— Cần truyền file descriptor qua SCM_RIGHTS
Pattern: daemon nhận lệnh từ nhiều client với select()
daemon_server.c — non-blocking, select(), pattern chuẩn embedded
#include <sys/select.h>
#include <sys/socket.h>
#include <sys/un.h>
#include <unistd.h>
#include <signal.h>
#include <string.h>
#include <stdio.h>
#include <errno.h>

#define SOCK_PATH  "/var/run/myapp/control.sock"
#define MAX_CLIENTS 8

static volatile int running = 1;
void on_term(int s) { (void)s; running = 0; }

int main(void) {
    signal(SIGTERM, on_term);
    signal(SIGINT,  on_term);
    signal(SIGPIPE, SIG_IGN);  /* Bắt buộc: tránh crash khi client thoát */

    int srv = socket(AF_UNIX,
                     SOCK_STREAM | SOCK_NONBLOCK | SOCK_CLOEXEC, 0);
    struct sockaddr_un addr = { .sun_family = AF_UNIX };
    strncpy(addr.sun_path, SOCK_PATH, sizeof(addr.sun_path)-1);
    unlink(SOCK_PATH);
    bind(srv, (struct sockaddr *)&addr, sizeof(addr));
    listen(srv, MAX_CLIENTS);

    int clients[MAX_CLIENTS];
    for (int i = 0; i < MAX_CLIENTS; i++) clients[i] = -1;

    printf("Daemon ready: %s\n", SOCK_PATH);

    while (running) {
        /* Xây fd_set mỗi vòng lặp */
        fd_set rfds;
        FD_ZERO(&rfds);
        FD_SET(srv, &rfds);
        int maxfd = srv;

        for (int i = 0; i < MAX_CLIENTS; i++) {
            if (clients[i] != -1) {
                FD_SET(clients[i], &rfds);
                if (clients[i] > maxfd) maxfd = clients[i];
            }
        }

        struct timeval tv = { .tv_sec = 1 };  /* Timeout 1s → kiểm tra running */
        int ready = select(maxfd + 1, &rfds, NULL, NULL, &tv);
        if (ready == -1) {
            if (errno == EINTR) continue;  /* Signal ngắt, bình thường */
            break;
        }

        /* Có client mới kết nối? */
        if (FD_ISSET(srv, &rfds)) {
            int cli = accept4(srv, NULL, NULL, SOCK_CLOEXEC);
            if (cli != -1) {
                for (int i = 0; i < MAX_CLIENTS; i++) {
                    if (clients[i] == -1) { clients[i] = cli; break; }
                }
            }
        }

        /* Đọc từ các client đang kết nối */
        for (int i = 0; i < MAX_CLIENTS; i++) {
            if (clients[i] == -1 || !FD_ISSET(clients[i], &rfds))
                continue;

            char buf[128];
            ssize_t n = recv(clients[i], buf, sizeof(buf)-1, 0);
            if (n <= 0) {
                /* n==0: client đóng bình thường; n==-1: lỗi */
                close(clients[i]);
                clients[i] = -1;
            } else {
                buf[n] = '\0';
                printf("Lenh tu client[%d]: %s\n", i, buf);
                send(clients[i], "OK", 2, MSG_NOSIGNAL);
            }
        }
    }

    close(srv);
    unlink(SOCK_PATH);
    printf("Daemon dung.\n");
    return 0;
}
Ràng buộc thực tế trên embedded
sun_path tối đa 108 ký tự: Giới hạn của struct sockaddr_un trên Linux. Path dài hơn bị cắt silently khi strncpy. Thường dùng /var/run/app/x.sock hoặc /tmp/x.sock.
/var/run cần mount tmpfs: Trên embedded rootfs read-only, /var/run cần được mount tmpfs trong init script. Nếu không, bind() lỗi EROFS. Kiểm tra: mount | grep tmpfs
unlink() khi khởi động lại: Nếu daemon crash mà không cleanup, socket file còn trên FS. Lần sau, bind() lỗi EADDRINUSE. Luôn unlink(path) trước bind() — pattern bắt buộc.
Permission socket file: Socket file kế thừa umask. Nếu client chạy user khác cần dùng chmod() sau bind(). Hoặc dùng abstract namespace để tránh vấn đề permission hoàn toàn.
Abstract namespace — không có file trên filesystem
Abstract socket — tự cleanup khi tất cả fd đóng, không cần unlink

Bước tiếp theo: Sau Unix Socket, IPC tiếp theo nên học là Shared Memory (mmap + POSIX shm) — throughput cao nhất vì zero-copy, cần thiết khi truyền dữ liệu lớn như frame ảnh từ camera hoặc buffer audio. Sau đó là Message Queue cho pattern publish-subscribe.
Chủ đề nâng cao
SCM_RIGHTS truyền fd, SO_PEERCRED xác thực client, hiệu năng so với TCP loopback, SOCK_SEQPACKET thực chiến.
SCM_RIGHTS — truyền file descriptor giữa các process
Đây là khả năng duy nhất của Unix Socket mà không IPC nào khác có: truyền một file descriptor đang mở từ process A sang process B. Process B nhận được một fd mới trỏ đến cùng file/socket/device mà A đang mở.

Ứng dụng thực tế:
— Process cha mở device (/dev/spi0), truyền fd cho process con không có quyền root
— Supervisor mở log file, truyền fd cho worker process (worker không cần biết path)
— Web server nhận connection, truyền client fd sang worker process để xử lý

Cơ chế: dùng sendmsg() với struct msghdr chứa ancillary data (struct cmsghdr) có type SCM_RIGHTS. Kernel sẽ duplicate fd sang process nhận.
send_fd() — gửi 1 file descriptor qua Unix Socket
#include <sys/socket.h>
#include <string.h>

/* Gửi fd_to_send qua socket 'sock'. Trả về 0 thành công, -1 lỗi */
int send_fd(int sock, int fd_to_send) {
    /* Cần gửi ít nhất 1 byte data thật — nếu không sendmsg sẽ lỗi */
    char dummy = 'F';
    struct iovec iov = { .iov_base = &dummy, .iov_len = 1 };

    /* Buffer cho ancillary data — chứa 1 int (fd) */
    char cmsg_buf[CMSG_SPACE(sizeof(int))];
    memset(cmsg_buf, 0, sizeof(cmsg_buf));

    struct msghdr msg = {
        .msg_iov     = &iov,
        .msg_iovlen  = 1,
        .msg_control    = cmsg_buf,
        .msg_controllen = sizeof(cmsg_buf)
    };

    /* Điền ancillary header */
    struct cmsghdr *cmsg  = CMSG_FIRSTHDR(&msg);
    cmsg->cmsg_level = SOL_SOCKET;
    cmsg->cmsg_type  = SCM_RIGHTS;       /* "Tôi đang gửi file descriptor" */
    cmsg->cmsg_len   = CMSG_LEN(sizeof(int));
    *(int *)CMSG_DATA(cmsg) = fd_to_send; /* fd thực sự */

    return (sendmsg(sock, &msg, 0) == -1) ? -1 : 0;
}
recv_fd() — nhận file descriptor từ process khác
/* Nhận fd từ socket 'sock'. Trả về fd nhận được, hoặc -1 nếu lỗi */
int recv_fd(int sock) {
    char dummy;
    struct iovec iov = { .iov_base = &dummy, .iov_len = 1 };

    char cmsg_buf[CMSG_SPACE(sizeof(int))];

    struct msghdr msg = {
        .msg_iov        = &iov,
        .msg_iovlen     = 1,
        .msg_control    = cmsg_buf,
        .msg_controllen = sizeof(cmsg_buf)
    };

    if (recvmsg(sock, &msg, 0) == -1) return -1;

    struct cmsghdr *cmsg = CMSG_FIRSTHDR(&msg);
    if (cmsg == NULL ||
        cmsg->cmsg_level != SOL_SOCKET ||
        cmsg->cmsg_type  != SCM_RIGHTS) {
        return -1;   /* Không phải SCM_RIGHTS message */
    }

    int received_fd;
    memcpy(&received_fd, CMSG_DATA(cmsg), sizeof(int));
    return received_fd;

    /* Lưu ý: fd nhận được là số MỚI trong process nhận,
       có thể khác số fd ở process gửi. Nhưng trỏ cùng file. */
}
Sử dụng thực tế: Process A mở /dev/i2c-1 với quyền root, rồi gửi fd qua Unix Socket cho process B chạy non-root. Process B dùng fd này để đọc/ghi I2C trực tiếp mà không cần quyền root. Đây là pattern privilege separation rất phổ biến trong embedded security.
SO_PEERCRED — xác thực client là ai
Khi server accept một kết nối, nó có thể hỏi kernel: "Client này là process nào, chạy với user/group gì?" — mà client không thể giả mạo thông tin này (kernel tự cung cấp).

Rất quan trọng trong embedded khi server daemon cần từ chối lệnh từ process không đủ quyền.

SOCK_SEQPACKET — code ví dụ thực chiến
SOCK_SEQPACKET kết hợp sự tin cậy của STREAM với message boundary của DGRAM. Không cần thiết kế framing protocol. Rất phù hợp để gửi struct command/response.

/* Cả server và client đều dùng struct này */
typedef struct {
    uint8_t  cmd;       /* 0=PING, 1=SET_LED, 2=GET_TEMP */
    uint8_t  reserved;
    uint16_t payload;
} Command;

typedef struct {
    uint8_t  status;    /* 0=OK, 1=ERROR */
    uint8_t  reserved;
    uint16_t value;
} Response;

/* === Server side === */
int srv = socket(AF_UNIX, SOCK_SEQPACKET | SOCK_CLOEXEC, 0);
/* bind(), listen() giống SOCK_STREAM */

int cli = accept4(srv, NULL, NULL, SOCK_CLOEXEC);

Command cmd;
ssize_t n = recv(cli, &cmd, sizeof(cmd), 0);
/* n == sizeof(Command) LUÔN ĐÚNG nếu client gửi đúng 1 struct.
   Không bao giờ nhận nửa struct — kernel đảm bảo message boundary.
   Nếu n == 0 → client đóng kết nối */

if (n == sizeof(cmd)) {
    Response resp = { .status = 0, .value = 42 };
    send(cli, &resp, sizeof(resp), MSG_NOSIGNAL);
    /* Client nhận đúng sizeof(Response), không hơn không kém */
}

/* === Client side === */

/* connect() giống SOCK_STREAM */

   - Không cần header length prefix

Đồng bộ hay Bất đồng bộ? — Socket nhận dữ liệu thế nào
Mặc định là blocking (đồng bộ). Có 4 cách chuyển sang bất đồng bộ — signal chỉ là 1 trong số đó và ít dùng nhất.
Mặc định: Blocking I/O (đồng bộ)
Khi gọi recv(fd, buf, len, 0), kernel sẽ block process — process chuyển sang trạng thái SLEEP và không chiếm CPU. Kernel đánh dấu "process này đang chờ dữ liệu trên fd này". Khi data đến, kernel đánh thức process, copy data vào buf, recv() return.

Tương tự cho accept(): nếu chưa có client nào connect, accept() block cho đến khi có.

Đây là cách đơn giản nhất và đủ dùng cho: single-client, protocol request-response tuần tự, embedded đơn giản.
Blocking ≠ lãng phí CPU. Process đang block không chiếm CPU — kernel chuyển nó sang hàng đợi SLEEP. CPU chạy process khác. Chỉ khi data đến, kernel mới đánh thức process. Đây là thiết kế hiệu quả, không phải "chờ bận" (busy-wait).

Cách 1: Non-blocking — O_NONBLOCK
Đặt socket ở chế độ non-blocking: mọi operation trả về ngay lập tức. Nếu chưa có data, recv() trả -1 với errno = EAGAIN. Nếu accept queue rỗng, accept() cũng trả -1/EAGAIN.

Hai cách bật:
— Khi tạo socket: socket(AF_UNIX, SOCK_STREAM | SOCK_NONBLOCK, 0)
— Sau khi tạo: fcntl(fd, F_SETFL, O_NONBLOCK)
Non-blocking recv — phải tự kiểm tra EAGAIN
#include <fcntl.h>
#include <errno.h>

/* Cách 1: bật khi tạo socket */
int fd = socket(AF_UNIX, SOCK_STREAM | SOCK_NONBLOCK | SOCK_CLOEXEC, 0);

/* Cách 2: bật sau bằng fcntl */
int flags = fcntl(fd, F_GETFL, 0);
fcntl(fd, F_SETFL, flags | O_NONBLOCK);

/* Đọc dữ liệu — không block */
char buf[256];
ssize_t n = recv(fd, buf, sizeof(buf), 0);

if (n > 0) {
    /* Có data → xử lý */
    buf[n] = '\0';
    printf("Nhận: %s\n", buf);
} else if (n == 0) {
    /* Peer đóng kết nối */
    printf("Client disconnected\n");
} else {  /* n == -1 */
    if (errno == EAGAIN || errno == EWOULDBLOCK) {
        /* CHƯA có data — bình thường, không phải lỗi */
        /* Làm việc khác rồi thử lại sau */
    } else {
        perror("recv");  /* Lỗi thật */
    }
}
Không dùng non-blocking đơn lẻ với busy-wait loop!
while (1) { n = recv(...); if (n == -1 && errno == EAGAIN) continue; }
Vòng lặp này chạy liên tục 100% CPU mà không nhận được gì. Trên embedded, pin hết nhanh, CPU nóng. Luôn kết hợp non-blocking với select/poll/epoll hoặc thêm usleep() — nhưng select/epoll là cách đúng.
Cách 2: select() / poll() — I/O Multiplexing ★ phổ biến nhất
select() cho phép process block chờ nhiều fd cùng lúc. Khi bất kỳ fd nào có data (hoặc timeout), select() return và cho biết fd nào sẵn sàng.

Cơ chế kernel: process gọi select() → kernel đặt process vào wait queue của TẤT CẢ các fd → khi data đến trên bất kỳ fd nào → kernel đánh thức process → select() return.

Đây là cách server phục vụ nhiều client mà chỉ cần 1 thread. Đã dùng trong ví dụ daemon ở tab "6. Embedded Linux".
poll() — đơn giản hơn select(), không giới hạn 1024 fd
#include <poll.h>

/* Theo dõi server socket + 1 client socket cùng lúc */
struct pollfd fds[2];
fds[0].fd     = srv_fd;    /* server socket — chờ client mới */
fds[0].events = POLLIN;    /* POLLIN = có data đọc được hoặc có connect mới */
fds[1].fd     = cli_fd;    /* client đã kết nối — chờ data */
fds[1].events = POLLIN;

while (running) {
    /* Block tối đa 1000ms, hoặc tỉnh ngay khi có sự kiện */
    int ready = poll(fds, 2, 1000);

    if (ready == -1) {
        if (errno == EINTR) continue;  /* signal ngắt */
        break;
    }
    if (ready == 0) {
        /* Timeout — không có sự kiện nào trong 1s */
        continue;
    }

    /* Kiểm tra từng fd */
    if (fds[0].revents & POLLIN) {
        /* Có client mới connect → accept() sẽ KHÔNG block */
        int new_cli = accept4(srv_fd, NULL, NULL, SOCK_CLOEXEC);
        /* ... */
    }
    if (fds[1].revents & POLLIN) {
        /* Client gửi data → recv() sẽ KHÔNG block */
        ssize_t n = recv(cli_fd, buf, sizeof(buf), 0);
        /* ... */
    }
    if (fds[1].revents & (POLLHUP | POLLERR)) {
        /* Client đóng hoặc lỗi */
        close(cli_fd);
    }
}

/* POLLIN  = có data đọc được (hoặc accept ready)
   POLLOUT = có thể ghi mà không block
   POLLHUP = peer đóng kết nối
   POLLERR = lỗi trên fd */
select() vs poll(): Cả hai làm cùng 1 việc. select() giới hạn FD_SETSIZE (thường 1024) fd và phải rebuild fd_set mỗi lần gọi. poll() không giới hạn số fd và API sạch hơn. Trên embedded, poll() được khuyên dùng hơn select() cho code mới.
Cách 3: epoll — hiệu năng cao nhất (Linux-only)
select/poll mỗi lần gọi phải quét toàn bộ danh sách fd — O(n). Với 1000 client mà chỉ 5 client có data, vẫn phải kiểm tra 1000 fd.

epoll dùng cơ chế callback trong kernel: khi data đến trên fd nào, kernel thêm fd đó vào "ready list". epoll_wait() chỉ trả về những fd thực sự có sự kiện — O(1) per event.

Trên embedded, epoll quan trọng khi daemon cần xử lý nhiều nguồn dữ liệu (socket + serial + GPIO + timer) trên 1 thread.
epoll — server socket + timerfd trên cùng event loop
#include <sys/epoll.h>

int epfd = epoll_create1(EPOLL_CLOEXEC);

/* Đăng ký server socket */
struct epoll_event ev;
ev.events  = EPOLLIN;       /* Quan tâm: có client mới */
ev.data.fd = srv_fd;
epoll_ctl(epfd, EPOLL_CTL_ADD, srv_fd, &ev);

/* Đăng ký thêm client fd sau khi accept */
/* ev.data.fd = cli_fd;
   epoll_ctl(epfd, EPOLL_CTL_ADD, cli_fd, &ev); */

/* Event loop */
struct epoll_event events[16];  /* Nhận tối đa 16 sự kiện/lần */

while (running) {
    /* Block tối đa 1000ms */
    int nfds = epoll_wait(epfd, events, 16, 1000);

    for (int i = 0; i < nfds; i++) {
        if (events[i].data.fd == srv_fd) {
            /* Client mới → accept */
            int cli = accept4(srv_fd, NULL, NULL, SOCK_CLOEXEC);
            if (cli != -1) {
                ev.events  = EPOLLIN;
                ev.data.fd = cli;
                epoll_ctl(epfd, EPOLL_CTL_ADD, cli, &ev);
            }
        } else {
            /* Data từ client */
            char buf[256];
            ssize_t n = recv(events[i].data.fd, buf, sizeof(buf), 0);
            if (n <= 0) {
                /* Client đóng hoặc lỗi → gỡ khỏi epoll */
                epoll_ctl(epfd, EPOLL_CTL_DEL,
                          events[i].data.fd, NULL);
                close(events[i].data.fd);
            } else {
                buf[n] = '\0';
                printf("Nhận: %s\n", buf);
                send(events[i].data.fd, "OK", 2, MSG_NOSIGNAL);
            }
        }
    }
}
close(epfd);

/* Ưu điểm epoll trên embedded:
   - 1 thread xử lý socket + serial + GPIO + timerfd
   - Không cần tạo nhiều thread → tiết kiệm RAM
   - O(1) per event → hiệu quả với nhiều fd */
Cách 4: SIGIO — bắn signal khi có data
Đúng, kernel CÓ THỂ bắn signal khi có data — cơ chế gọi là Signal-driven I/O. Khi data đến trên socket, kernel gửi SIGIO (hoặc SIGPOLL) đến process. Process đang làm việc khác sẽ bị ngắt, chạy signal handler, rồi quay lại công việc.

Nhưng trong thực tế gần như không ai dùng. Lý do:
Thiết lập SIGIO — hoạt động nhưng KHÔNG khuyên dùng
#include <signal.h>
#include <fcntl.h>

static volatile int data_ready = 0;

void sigio_handler(int sig) {
    (void)sig;
    data_ready = 1;
    /* CẢNH BÁO: trong signal handler chỉ được gọi
       async-signal-safe functions!
       KHÔNG được gọi: printf, malloc, recv, send, mutex...
       Chỉ set flag rồi return */
}

int main(void) {
    int fd = socket(AF_UNIX, SOCK_STREAM | SOCK_CLOEXEC, 0);
    /* ... connect() ... */

    /* Bước 1: Đăng ký signal handler */
    struct sigaction sa;
    sa.sa_handler = sigio_handler;
    sa.sa_flags = 0;
    sigemptyset(&sa.sa_mask);
    sigaction(SIGIO, &sa, NULL);

    /* Bước 2: Đặt process owner của fd */
    fcntl(fd, F_SETOWN, getpid());

    /* Bước 3: Bật async notification */
    int flags = fcntl(fd, F_GETFL, 0);
    fcntl(fd, F_SETFL, flags | O_ASYNC | O_NONBLOCK);

    /* Bây giờ khi data đến → kernel gửi SIGIO → handler chạy */

    while (running) {
        if (data_ready) {
            data_ready = 0;
            char buf[256];
            ssize_t n = recv(fd, buf, sizeof(buf), 0);
            if (n > 0) {
                buf[n] = '\0';
                printf("Nhận: %s\n", buf);
            }
        }
        /* Làm việc khác... */
        do_other_work();
    }
}
Tại sao SIGIO gần như không dùng?
1. Chỉ 1 fd per process (traditional SIGIO): Nếu có 10 socket, SIGIO chỉ cho biết "có data ở đâu đó" — không biết fd nào! Phải kiểm tra tất cả. F_SETSIG + sigqueue cải thiện nhưng phức tạp.

2. Signal handler hạn chế nghiêm trọng: Trong handler chỉ được gọi async-signal-safe functions. Không được printf, malloc, recv, không lock mutex. Chỉ có thể set flag.

3. Race condition: Signal có thể đến giữa lúc kiểm tra flag → mất signal. Cần pselect() hoặc signalfd() để xử lý đúng.

4. Khó debug: Signal handler chạy asynchronous, breakpoint không hoạt động bình thường, timing-dependent bug.

Kết luận: Dùng epoll thay SIGIO trong mọi trường hợp. Nếu muốn signal-style nhưng an toàn, dùng signalfd() — biến signal thành fd, đọc bằng read() bình thường.
signalfd() — signal nhưng an toàn
signalfd() biến signal thành file descriptor — đọc bằng read() bình thường, có thể kết hợp với epoll. Không cần signal handler, không có race condition.

Ứng dụng: đặt SIGTERM/SIGINT vào epoll cùng socket → 1 event loop xử lý cả signal lẫn data.
signalfd + epoll — xử lý signal cùng event loop
#include <sys/signalfd.h>
#include <sys/epoll.h>
#include <signal.h>

/* Block signal truyền thống — bắt buộc khi dùng signalfd */
sigset_t mask;
sigemptyset(&mask);
sigaddset(&mask, SIGTERM);
sigaddset(&mask, SIGINT);
sigprocmask(SIG_BLOCK, &mask, NULL);  /* Block để signal không kill process */

/* Tạo signalfd — đọc signal bằng read() */
int sig_fd = signalfd(-1, &mask, SFD_CLOEXEC);

/* Thêm vào epoll cùng socket */
struct epoll_event ev;
ev.events  = EPOLLIN;
ev.data.fd = sig_fd;  /* fd này sẽ readable khi có SIGTERM/SIGINT */
epoll_ctl(epfd, EPOLL_CTL_ADD, sig_fd, &ev);

/* Trong event loop */
for (int i = 0; i < nfds; i++) {
    if (events[i].data.fd == sig_fd) {
        /* Signal đến — đọc như data bình thường */
        struct signalfd_siginfo info;
        read(sig_fd, &info, sizeof(info));
        printf("Nhận signal %d, thoát...\n", info.ssi_signo);
        running = 0;
    } else if (events[i].data.fd == srv_fd) {
        /* Client mới connect */
    } else {
        /* Data từ client */
    }
}

   - Socket server (client mới)
   - Socket client (data)
   - Signal (shutdown gracefully)
   - timerfd (periodic task)
   - eventfd (thread notification)
   Tất cả đều là fd → tất cả đều vào epoll */

Tra cứu macro — không cần nhớ, chỉ cần biết tìm ở đâu
Macro (biến IN HOA) là hằng số được định nghĩa trong header file. Không ai nhớ hết — cách làm việc đúng là biết tra cứu nhanh.

# 1. man page — nguồn chính xác nhất, có giải thích đầy đủ
man 2 socket        # AF_UNIX, SOCK_STREAM, SOCK_CLOEXEC...
man 2 send          # MSG_NOSIGNAL, MSG_DONTWAIT, MSG_PEEK...
man 2 recv          # MSG_WAITALL, MSG_PEEK...
man 2 poll          # POLLIN, POLLOUT, POLLHUP...

man 3 errno         # EAGAIN, EPIPE, EADDRINUSE...
man 2 fcntl         # F_SETFL, O_NONBLOCK, O_ASYNC...

# 2. Tìm định nghĩa trong header file
grep -rn "MSG_NOSIGNAL" /usr/include/
