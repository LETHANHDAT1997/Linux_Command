Khi bạn gọi open("/etc/passwd", O_RDONLY), Linux trải qua nhiều tầng trừu tượng trước khi thực sự đọc dữ liệu từ đĩa. Dưới đây là kiến trúc tổng thể:

💡
Ý tưởng cốt lõi

VFS là "bộ phiên dịch" cho phép Linux xử lý mọi loại filesystem (ext4, FAT, NFS, procfs…) bằng một giao diện thống nhất. Application không cần biết file nằm ở ext4 hay NFS.

1
Ổ cứng, Phân vùng & Block
Nền tảng vật lý — nơi dữ liệu thực sự lưu trú

1.1 Block Device là gì?

Ổ cứng (HDD/SSD) là block device — thiết bị đọc/ghi dữ liệu theo từng khối (block) cố định, thường là 512 byte (sector) hoặc 4096 byte (4K sector / logical block).

🖴 Ổ cứng vật lý
┌────────────────────────────────────────────────────────────┐ │ /dev/sda (toàn bộ đĩa) │ │ ┌───────────┐ ┌─────────────────────┐ ┌──────────────┐ │ │ │ MBR/GPT │ │ /dev/sda1 (boot) │ │ /dev/sda2 │ │ │ │ 512 byte │ │ ext4, 500MB │ │ ext4, 50GB │ │ │ └───────────┘ └─────────────────────┘ └──────────────┘ │ │ │ │ Mỗi phân vùng gồm hàng nghìn BLOCK (4096 byte mỗi block) │ └────────────────────────────────────────────────────────────┘

Block4096 B (ext4 mặc định)Đơn vị FS quản lý. Block = N sector

1.2 Cấu trúc Ext4 trên đĩa

Ext4 (filesystem phổ biến nhất trên Linux) chia phân vùng thành các Block Group:

PHÂN VÙNG /dev/sda2 (ext4) ┌──────────┬─────────────────────────────────────────────────────────┐ │ Block 0 │ Superblock (siêu khối): kích thước FS, số inode, │ │ │ số block tự do, magic number 0xEF53, … │ ├──────────┼─────────────────────────────────────────────────────────┤ │ Group 0 │ ┌──────────┬───────────┬──────────┬────────────────┐ │ │ │ │ Group │ Block │ Inode │ Data Blocks │ │ │ │ │ Descr. │ Bitmap │ Table │ (dữ liệu thực)│ │ │ │ └──────────┴───────────┴──────────┴────────────────┘ │ ├──────────┼─────────────────────────────────────────────────────────┤ │ Group 1 │ ...tương tự... │ ├──────────┼─────────────────────────────────────────────────────────┤ │ ... │ │ └──────────┴─────────────────────────────────────────────────────────┘
⚠️
Quan trọng: Block ≠ File

Một file không nhất thiết chiếm các block liên tiếp. FS dùng inode để ánh xạ "các block nào thuộc file này". Đây là lý do file bị phân mảnh (fragmentation) sau nhiều lần ghi/xóa.

Inode — Trái tim của mọi tệp tin
Index Node: cấu trúc lưu trữ metadata và con trỏ đến dữ liệu

2.1 Inode là gì?

Mỗi file/thư mục trên Linux tương ứng với một inode. Inode là một cấu trúc dữ liệu trên đĩa lưu mọi thứ về file — ngoại trừ tên file và nội dung.

🔑
Câu hỏi quan trọng

Tại sao inode không lưu tên file? Vì tên file được lưu trong thư mục cha (directory entry / dentry). Điều này cho phép một file có nhiều tên (hard link) cùng trỏ vào một inode.

2.3 Cơ chế ánh xạ Block — Ext2/3 cũ

Ext2/3 dùng sơ đồ indirect block kinh điển để file có thể lớn hơn 12 block:

INODE → DATA BLOCKS inode ┌──────────────────┐ │ i_block[0] ─────┼──→ Block 100 ← dữ liệu trực tiếp (0–11: 48KB) │ i_block[1] ─────┼──→ Block 215 │ ... │ │ i_block[11] ─────┼──→ Block 998 │ │ │ i_block[12] ─────┼──→ ┌─────────────────────────┐ indirect │ (indirect) │ │ ptr→100 ptr→200 ptr→305 │──→ blocks │ │ └─────────────────────────┘ (thêm 4MB) │ i_block[13] ─────┼──→ double indirect → thêm 4GB │ i_block[14] ─────┼──→ triple indirect → thêm 4TB └──────────────────┘

Ext4 cải tiến bằng Extents — thay vì liệt kê từng block, lưu dải block liên tiếp (block_start, length), giúp file lớn nhanh hơn và ít bị phân mảnh hơn.

2.4 Inode Number & Thư mục

Mỗi inode có một số inode (inode number) duy nhất trong filesystem. Thư mục (directory) thực ra là file đặc biệt chứa danh sách (tên_file → inode_number):

NỘI DUNG THƯ MỤC /home/dat/ ┌──────────────────────────────────────────────┐ │ Tên entry │ Inode số │ Loại │ ├────────────────┼─────────────┼───────────────┤ │ "." │ 4521 │ DIR (chính nó)│ │ ".." │ 43 │ DIR (cha) │ │ "main.c" │ 7809 │ REG │ │ "build" │ 7812 │ DIR │ │ "firmware.bin"│ 7815 │ REG │ └──────────────────────────────────────────────┘

Khi bạn gõ ls -i, cột đầu tiên chính là inode number:

$ ls -i /home/dat/
7809 main.c    7812 build/    7815 firmware.bin

$ stat main.c
  File: main.c
  Size: 4096        Inode: 7809       Links: 1
Access: (0644/-rw-r--r--)  Uid: 1000  Gid: 1000

2.5 Hard Link vs Soft Link

🔗 Hard Link cùng inode

Tạo thêm một tên khác trỏ vào cùng một inode. Xóa tên này, inode vẫn còn nếu còn link khác. File bị xóa thật sự khi i_links_count == 0 và không có process nào đang mở.

ln main.c main_backup.c

🔀 Soft Link (Symlink) inode riêng

Tạo inode mới, loại S_IFLNK, nội dung là đường dẫn đến target. Xóa target → symlink bị "đứt" (dangling). Có thể link cross-filesystem.

ln -s /etc/nginx/nginx.conf nginx.conf

VFS & VNode
Virtual File System — lớp trừu tượng cho phép Linux "nói chuyện" với mọi loại FS

3.1 Vì sao cần VFS?

Linux hỗ trợ hàng chục filesystem: ext4, FAT32, NTFS, NFS, procfs, sysfs, tmpfs… Mỗi loại có cấu trúc đĩa khác nhau. Không có VFS, mỗi application phải biết mình đang dùng FS nào — điều này là không thực tế.

Application: open("/proc/cpuinfo")
↓
VFS: Tìm VNode cho đường dẫn này
↓
Dispatch đến FS driver tương ứng: procfs
↓
procfs tạo nội dung "on-the-fly" (không có block thật)

3.2 Bốn đối tượng chính trong VFS Linux

StructVai tròTương tự đĩa

struct fileĐại diện cho 1 lần mở file (có vị trí đọc/ghi)Không có trên đĩa (RAM only)

3.3 VNode là gì?

Thuật ngữ VNode (Virtual Node) xuất phát từ BSD Unix và được dùng để chỉ khái niệm trừu tượng của "một node trong VFS" — tức là đại diện trong bộ nhớ cho bất kỳ đối tượng nào trong cây filesystem (file, thư mục, device, socket…).

📌
VNode trong Linux kernel

Linux không có struct tên là vnode. Thay vào đó, Linux dùng struct inode làm VNode tương đương — nó nằm trong RAM và có con trỏ đến inode_operations (bảng hàm ảo theo FS cụ thể). Đây là cài đặt C của polymorphism trong OOP.

3.4 struct inode (VFS layer — trong RAM)

/* include/linux/fs.h (rút gọn) */
struct inode {
    umode_t         i_mode;       /* permissions & type */
    kuid_t          i_uid;
    kgid_t          i_gid;
    unsigned long   i_ino;        /* inode number */
    loff_t          i_size;       /* file size in bytes */
    unsigned int    i_nlink;      /* hard link count */
    atomic_t        i_count;      /* reference count */

    const struct inode_operations  *i_op;   /* vtable: lookup, create, link… */
    const struct file_operations   *i_fop;  /* vtable: read, write, mmap… */
    struct super_block             *i_sb;   /* filesystem này thuộc về */
    struct address_space           *i_mapping; /* page cache */
    /* ... */
};

3.5 Dentry Cache — tăng tốc path resolution

Mỗi khi kernel phân giải đường dẫn /home/dat/main.c, nó phải đi qua từng thành phần: / → home → dat → main.c. Để tránh đọc đĩa mỗi lần, kernel cache các bước này trong Dentry Cache (dcache):

dcache ┌─────────────────────────────────────────────────────┐ │ dentry("/") → inode #2 (root dir) │ │ dentry("home") → inode #43 │ │ dentry("dat") → inode #4521 │ │ dentry("main.c") → inode #7809 ← HIT! (cached)│ └─────────────────────────────────────────────────────┘

3.6 struct file — một lần mở cụ thể

struct file được tạo mỗi lần open() được gọi. Nó lưu trạng thái của lần mở đó (vị trí đọc/ghi, flags):

struct file {
    struct path          f_path;    /* dentry + vfsmount */
    struct inode        *f_inode;   /* inode của file */
    const struct file_operations *f_op;
    loff_t               f_pos;    /* vị trí đọc/ghi hiện tại */
    unsigned int         f_flags;  /* O_RDONLY, O_WRONLY… */
    atomic_long_t        f_count;  /* reference count */
    /* ... */
};
✅
Phân biệt inode và file

Nếu 2 process cùng open() một file, sẽ có 2 struct file khác nhau (2 vị trí đọc độc lập) nhưng chỉ có 1 inode. Ngược lại, nếu một process dùng dup(fd), 2 fd sẽ dùng chung một struct file (cùng vị trí đọc).

File Descriptor
Con số nhỏ bé mà application dùng để "nói chuyện" với kernel

4.1 File Descriptor là gì?

File Descriptor (fd) là một số nguyên không âm (0, 1, 2, 3, …) đại diện cho một tài nguyên đang mở trong tiến trình hiện tại. Nó là chỉ số vào bảng file của process.

💡
fd mặc định

Mỗi process sinh ra đã có sẵn 3 fd: 0 = stdin, 1 = stdout, 2 = stderr. Khi bạn open() một file, kernel cấp fd nhỏ nhất còn trống, bắt đầu từ 3.

4.2 Ba bảng của kernel

Kernel quản lý file qua 3 bảng lồng nhau:

PROCESS A KERNEL DISK ┌──────────────────┐ ┌──────────────────────┐ ┌─────────────────┐ │ fd table │ │ Open File Table │ │ Inode Table │ │ (per-process) │ │ (system-wide) │ │ (on disk/cache)│ │ │ │ │ │ │ │ fd 0 (stdin) ───┼──→ │ entry A: f_pos=0 │ │ │ │ fd 1 (stdout)───┼──→ │ entry B: f_pos=0 │ │ │ │ fd 2 (stderr)───┼──→ │ entry C: f_pos=0 │ │ │ │ fd 3 ───────────┼──→ │ entry D: f_pos=1024 ├──→ │ inode #7809 │ │ fd 4 ───────────┼──╮ │ entry E: f_pos=0 ├──→ │ inode #7809 │ └──────────────────┘ │ └──────────────────────┘ └─────────────────┘ │ ↑ ┌──────────────────┐ ╰──→ (Process B dup()/fork()) │ PROCESS B │ │ fd 5 ───────────┼──→ entry E (cùng struct file) └──────────────────┘
BảngPhạm viLưu gì

4.3 Vòng đời của một File Descriptor

/* 1. Mở file: kernel tạo struct file, cấp fd */
int fd = open("/home/dat/main.c", O_RDONLY);
/* fd = 3 (ví dụ) */

/* 2. Đọc: kernel dùng fd để tìm struct file → inode → block */
char buf[128];
read(fd, buf, sizeof(buf));   /* f_pos tự động tăng */

/* 3. Lseek: thay đổi f_pos trong struct file */
lseek(fd, 0, SEEK_SET);       /* quay về đầu file */

/* 4. Đóng: struct file bị giải phóng, fd được tái sử dụng */
close(fd);

4.4 fd không chỉ dành cho file thông thường

Triết lý Unix: "Everything is a file". Các loại tài nguyên sau đây đều dùng fd:

📁 Regular File / Directory

File thông thường, thư mục trên đĩa

🖨️ Device File

/dev/sda, /dev/ttyUSB0, /dev/mem

🔌 Socket

TCP/UDP socket, Unix Domain Socket cũng là fd

📡 Pipe / FIFO

pipe() tạo 2 fd: đầu đọc và đầu ghi

⏰ timerfd / eventfd

Timer và event notification qua fd (Linux-specific)

🔔 signalfd / epoll

Signal và I/O multiplexing cũng qua fd

Vòng đời đầy đủ khi mở một file
Từ open() đến block đọc từ đĩa — step by step

5.1 Hành trình của open("/home/dat/main.c", O_RDONLY)

  • 1
    Syscall trap: Application gọi open() → CPU chuyển từ user mode sang kernel mode qua syscall số 2 (x86_64).
  • 2
    Path resolution: VFS phân giải /home/dat/main.c từng bước. Với mỗi thành phần, kernel tìm trong dcache trước, nếu miss mới đọc thư mục từ đĩa.
  • 3
    Permission check: Kernel kiểm tra i_mode và i_uid/i_gid trong inode so với uid/gid của process. Nếu không có quyền → EACCES.
  • 4
    Cấp struct file: Kernel cấp một struct file mới, khởi tạo f_pos = 0, f_flags = O_RDONLY, trỏ đến inode.
  • 5
    Cấp fd: Kernel chọn số nguyên nhỏ nhất còn trống trong fd table của process, ghi con trỏ đến struct file vừa tạo. Trả về số fd này về user space.
  • 6
    Return: CPU trở về user mode, open() trả về fd = 3.

5.2 Hành trình của read(fd, buf, 512)

  • 1
    Kernel dùng fd tra bảng → tìm ra struct file → lấy f_pos và con trỏ inode.
  • 2
    Page Cache lookup: Kernel kiểm tra xem dữ liệu tại offset f_pos đã có trong page cache chưa. Page cache là RAM cache dữ liệu đĩa.
  • 3
    Cache HIT: Dữ liệu đã có trong RAM → copy thẳng vào buf của user. f_pos += 512. Rất nhanh (ns).
  • 4
    Cache MISS: Kernel phải tìm block numbers trong inode (qua extents/indirect), gửi lệnh đọc đến Block Layer → Device Driver → đĩa vật lý. Dữ liệu trả về được nạp vào page cache rồi copy sang buf. Chậm hơn nhiều (μs–ms).
READ PATH read(fd, buf, 512) │ ↓ [kernel: fd → struct file → inode → offset] │ ↓ PAGE CACHE: offset ∈ cache? ├─ YES ──→ memcpy → buf (fast path: ~100ns) │ └─ NO ──→ Block layer: inode.extent → block# 4200 │ ↓ Device driver → NVMe/SATA controller │ ↓ 🖴 Đọc sector từ đĩa (~100μs NVMe, ~10ms HDD) │ ↓ Nạp vào page cache → memcpy → buf

Chủ đề Nâng Cao
Page Cache, mmap, O_DIRECT, inotify, và embedded considerations

6.1 Page Cache

Page Cache là bộ nhớ kernel dùng để cache nội dung file. Khi RAM còn trống, Linux ưu tiên dùng cho page cache. Khi RAM thiếu, kernel evict (đẩy ra) các page sạch (không bị dirty).

# Xem dung lượng page cache hiện tại
$ cat /proc/meminfo | grep -E "Cached|Buffers"
Buffers:          210432 kB   # block metadata cache
Cached:         5421988 kB   # page cache (file data)

# Flush page cache (testing only)
$ echo 3 > /proc/sys/vm/drop_caches

6.2 mmap — ánh xạ file vào địa chỉ ảo

mmap() ánh xạ file trực tiếp vào không gian địa chỉ ảo của process. Thay vì read()/write(), bạn truy cập file như truy cập mảng trong RAM:

int fd = open("firmware.bin", O_RDONLY);
void *addr = mmap(NULL, file_size, PROT_READ,
                   MAP_PRIVATE, fd, 0);

/* Truy cập như mảng — kernel load page khi cần (demand paging) */
uint32_t magic = *((uint32_t*)addr);

munmap(addr, file_size);
close(fd);
🚀
mmap zero-copy

mmap giúp tránh copy dữ liệu từ kernel space sang user space — dữ liệu ở page cache được ánh xạ trực tiếp vào address space của process. Rất hiệu quả cho file lớn như firmware image hoặc database.

6.3 O_DIRECT — bỏ qua Page Cache

Trong embedded / database systems, đôi khi bạn muốn quản lý cache tự mình, không dùng page cache của kernel:

int fd = open("/dev/sda", O_RDWR | O_DIRECT);
/* Buffer PHẢI căn chỉnh theo sector size (512 hoặc 4096 byte) */
void *buf;
posix_memalign(&buf, 4096, 4096);
read(fd, buf, 4096);  /* DMA thẳng từ đĩa vào buf, không qua cache */

6.4 inotify — giám sát thay đổi file

Linux cung cấp inotify API cho phép monitor sự kiện trên file/thư mục:

int ifd = inotify_init();
int wd  = inotify_add_watch(ifd, "/etc",
                IN_MODIFY | IN_CREATE | IN_DELETE);

/* Đọc events */
struct inotify_event ev;
read(ifd, &ev, sizeof(ev));
/* ev.name = tên file thay đổi, ev.mask = loại sự kiện */

⚠️
Embedded Consideration: fd leak

Trong embedded Linux (Buildroot), fd leak (mở file mà không close) là lỗi nghiêm trọng vì giới hạn fd thấp hơn desktop. Luôn kiểm tra return value của close() và dùng lsof / /proc/<pid>/fd để debug.

📚 Tóm tắt: Các khái niệm & mối quan hệ

  • Block — đơn vị lưu trữ vật lý nhỏ nhất mà FS quản lý (thường 4096 byte)
  • Inode (on-disk) — cấu trúc metadata trên đĩa: permissions, size, owner, con trỏ đến data blocks. Không chứa tên file.
  • Directory — file đặc biệt ánh xạ tên → inode number
  • Hard Link — nhiều tên trỏ vào cùng một inode. File tồn tại đến khi link count = 0.
  • VFS — tầng trừu tượng trong kernel cho phép xử lý mọi FS bằng interface thống nhất
  • VNode / struct inode (VFS) — đại diện in-memory của inode, có bảng hàm ảo (vtable) theo FS
  • struct dentry — cache một bước trong đường dẫn, tăng tốc path resolution
  • struct file — một lần mở file: vị trí đọc/ghi (f_pos), flags, con trỏ inode
  • File Descriptor — số nguyên trong process, chỉ vào struct file trong open file table của kernel
  • Page Cache — RAM cache dữ liệu đĩa, giúp read/write lặp lại rất nhanh
