Pipe — ống dẫn dữ liệu một chiều
Loại IPC đơn giản nhất. Hiểu Pipe là nền tảng để hiểu hết các IPC còn lại.
Ý tưởng cốt lõi
Hãy hình dung một cái ống nước thực sự: nước chỉ chảy một chiều, bạn đổ vào một đầu và lấy ra ở đầu kia theo đúng thứ tự (FIFO). Pipe trong Linux hoạt động y hệt như vậy với dữ liệu bytes.

Kernel tạo ra một vùng đệm (buffer) trong RAM — thường 64 KB trên Linux hiện đại. Một process ghi vào đầu này, process kia đọc từ đầu kia. Kernel hoàn toàn kiểm soát việc đồng bộ.
Pipe trông như thế nào trong bộ nhớ
Code đầy đủ, chạy được — có giải thích từng dòng
pipe_basic.c — compile: gcc pipe_basic.c -o pipe_basic
#include <unistd.h>    /* pipe(), read(), write(), fork(), close() */
#include <stdio.h>     /* printf(), perror() */
#include <stdlib.h>    /* exit() */
#include <string.h>    /* strlen() */
#include <sys/wait.h>  /* wait() */

int main(void) {
    int fds[2];   /* fds[0] = đầu đọc, fds[1] = đầu ghi */

    /* ── Bước 1: Tạo pipe ──────────────────────────────── */
    if (pipe(fds) == -1) {
        perror("pipe");   /* in lỗi dạng: "pipe: Too many open files" */
        exit(1);
    }
    /* Lúc này kernel đã tạo buffer 64KB trong RAM.
       fds[0] và fds[1] là 2 file descriptor trỏ vào buffer đó. */

    /* ── Bước 2: Fork để tạo child process ─────────────── */
    pid_t pid = fork();
    if (pid == -1) { perror("fork"); exit(1); }

    if (pid == 0) {
        /* ═══ Đây là CHILD PROCESS (writer) ═══ */

        close(fds[0]);   /* Đóng đầu đọc — child không cần đọc.
                            Quan trọng: nếu không đóng, reader sẽ
                            không bao giờ thấy EOF! */

        const char *msg = "Xin chao tu child process!";
        ssize_t n = write(fds[1], msg, strlen(msg));
        /* write() copy dữ liệu từ user space vào kernel buffer.
           Nếu buffer đầy → write() sẽ BLOCK cho đến khi có chỗ. */

        if (n == -1) { perror("write"); exit(1); }
        printf("[Child]  Da ghi %zd bytes\n", n);

        close(fds[1]);   /* Đóng write-end → kernel biết không còn
                            writer nào nữa → reader sẽ thấy EOF */
        exit(0);

    } else {
        /* ═══ Đây là PARENT PROCESS (reader) ═══ */

        close(fds[1]);   /* Đóng đầu ghi — parent không cần ghi.
                            Nếu không đóng: read() sẽ block mãi mãi
                            vì vẫn còn 1 fd ghi đang mở! */

        char buf[128] = {0};
        ssize_t n = read(fds[0], buf, sizeof(buf) - 1);
        /* read() copy từ kernel buffer vào buf.
           - Nếu buffer rỗng và writer còn sống → BLOCK (chờ dữ liệu)
           - Nếu tất cả write-end đã đóng     → trả về 0 (EOF)
           - Nếu có lỗi                        → trả về -1 */

        if (n == -1) { perror("read"); exit(1); }
        if (n == 0)  { printf("[Parent] EOF — child da dong pipe\n"); }
        else         { printf("[Parent] Nhan duoc: \"%s\"\n", buf); }

        close(fds[0]);

        int status;
        wait(&status);   /* Chờ child kết thúc, tránh zombie */
        printf("[Parent] Child thoat voi code %d\n",
               WEXITSTATUS(status));
    }
    return 0;
}

Thứ tự in ra có thể đảo: child/parent chạy song song. Nhưng nội dung message luôn đúng.
Bên trong kernel: pipe thực sự hoạt động ra sao
Những hành vi blocking, EOF, PIPE_BUF — tất cả đều có lý do rõ ràng.
4 quy tắc blocking của pipe
Quy tắc 1 read() block khi buffer rỗng
Nếu không có byte nào trong buffer và vẫn còn ít nhất 1 write-end đang mở, read() ngủ cho đến khi writer ghi gì đó. Kernel không tốn CPU trong lúc chờ — scheduler đưa process vào trạng thái TASK_INTERRUPTIBLE.
Quy tắc 2 read() trả về 0 (EOF) khi tất cả write-end đã đóng
Đây là cách reader biết "không còn dữ liệu nào nữa". Lỗi cực phổ biến: quên close(fds[1]) trong parent → reader block mãi mãi vì kernel thấy vẫn còn 1 fd ghi đang mở.
Quy tắc 3 write() block khi buffer đầy
Buffer mặc định 64 KB. Nếu reader chậm không đọc kịp và writer tiếp tục ghi, đến lúc buffer đầy, write() block cho đến khi reader đọc bớt ra. Điều này tạo ra cơ chế back-pressure tự nhiên.
Quy tắc 4 write() lên pipe không có reader → SIGPIPE
Nếu tất cả read-end đã đóng mà writer vẫn ghi, kernel gửi SIGPIPE đến writer. Mặc định process bị kill. Có thể ignore signal này và kiểm tra errno == EPIPE từ write().
Hiểu rõ về EOF (End of File) trong Pipe
Nhiều người lập trình hay lầm tưởng EOF là một ký tự đặc biệt (như \0 hoặc một ký tự ẩn nào đó) được ghi vào cuối luồng dữ liệu. Thực tế:

— EOF là một trạng thái (state): Đây là tín hiệu từ kernel báo rằng không còn nguồn ghi nào có thể đưa thêm dữ liệu vào pipe nữa.
— Cách kernel kích hoạt EOF: Khi tất cả các file descriptor ghi (write-end / fds[1]) liên kết với pipe này trên hệ thống đã đóng (kể cả đầu ghi của chính process đang đọc) VÀ bộ đệm trong RAM đã hết dữ liệu.
— Hành vi của read(): Khi gặp trạng thái này, hàm read() trả về 0 ngay lập tức thay vì block. Nhận giá trị 0 là cách chương trình nhận biết đã đọc hết file/pipe.
— Lỗi kinh điển: Nếu bạn quên gọi close(fds[1]) ở một process nào đó, kernel thấy "vẫn còn khả năng có dữ liệu ghi vào" nên read() sẽ block vô hạn để chờ thay vì trả về EOF (0).
PIPE_BUF và tính atomic
PIPE_BUF trên Linux = 4096 bytes (4 KB). Đây là ngưỡng quan trọng:

— Nếu write(n) với n ≤ PIPE_BUF: kernel đảm bảo ghi atomic — không bị xen lẫn với write() của process khác. An toàn với nhiều writer.
— Nếu n > PIPE_BUF: kernel có thể chia nhỏ và các write() khác nhau có thể bị xen kẽ → cần lock thủ công.
Kiểm tra PIPE_BUF và kích thước buffer thực tế
#include <limits.h>   /* PIPE_BUF */
#include <fcntl.h>    /* F_GETPIPE_SZ, F_SETPIPE_SZ */

printf("PIPE_BUF = %d bytes\n", PIPE_BUF);  /* Thường là 4096 */

int fds[2];
pipe(fds);

/* Đọc kích thước buffer hiện tại của pipe */
int sz = fcntl(fds[0], F_GETPIPE_SZ);
printf("Pipe buffer size = %d bytes\n", sz);  /* Thường 65536 = 64KB */

/* Tăng kích thước buffer (cần quyền hoặc process là owner) */
fcntl(fds[1], F_SETPIPE_SZ, 1024 * 1024);  /* Tăng lên 1 MB */
/* Trên embedded: /proc/sys/fs/pipe-max-size giới hạn giá trị tối đa */
Pipe với vòng lặp đọc — cách đúng
pipe_loop.c — đọc cho đến EOF, xử lý từng chunk
/* Reader đúng cách: đọc trong vòng lặp */
char buf[4096];
ssize_t n;

while ((n = read(fds[0], buf, sizeof(buf))) > 0) {
    /* n bytes đã đọc được, xử lý [buf, buf+n) */
    fwrite(buf, 1, n, stdout);
}
/* Sau vòng lặp: n == 0 nghĩa là EOF (tất cả writer đã đóng)
                 n == -1 nghĩa là lỗi, kiểm tra errno */
if (n == -1) perror("read");


/* ─── Sai lầm thường gặp ─────────────────────────────── */
/* SAI: đọc 1 lần, giả sử lấy hết dữ liệu               */
read(fds[0], buf, sizeof(buf));   /* Có thể nhận ít hơn! */

/* ĐÚNG: luôn đọc trong vòng lặp đến khi n == 0 hoặc lỗi */
Lý do phải đọc trong vòng lặp: read() được phép trả về ít hơn số byte bạn yêu cầu ngay cả khi chưa EOF. Ví dụ: write 1000 bytes nhưng read() có thể trả về 512, rồi 488. Đây là hành vi hợp lệ hoàn toàn.
API chi tiết — từng tham số, từng flag
Giải thích đầy đủ pipe(), pipe2(), read(), write() trong ngữ cảnh pipe.
pipe() — tạo anonymous pipe
#include <unistd.h>
int pipe(int pipefd[2]);
pipefd[2]
Output: kernel điền vào mảng 2 phần tử.
• pipefd[0] = file descriptor đầu đọc (read end)
• pipefd[1] = file descriptor đầu ghi (write end)
Cả hai là fd thông thường, dùng được với read/write/close/select/poll/epoll.
Trả về: 0 khi thành công, -1 khi lỗi (errno được set: EMFILE = hết fd, ENFILE = system limit)
pipe2() — tạo pipe với flags (Linux-specific)
#include <fcntl.h> /* O_CLOEXEC, O_NONBLOCK */ #include <unistd.h> int pipe2(int pipefd[2], int flags);
pipefd[2]
Giống pipe() — kernel điền đầu đọc/ghi.
flags
Kết hợp bằng OR (|) các giá trị sau:
O_CLOEXECTự động đóng fd khi process gọi exec(). Luôn dùng flag này để tránh fd rò rỉ sang chương trình con. Tương đương gọi fcntl(fd, F_SETFD, FD_CLOEXEC) nhưng atomic.
O_NONBLOCKBiến read/write thành non-blocking. Thay vì block, hàm trả về -1 với errno=EAGAIN khi không có dữ liệu (read) hoặc buffer đầy (write). Dùng khi kết hợp với event loop (select/poll/epoll).
Trả về: 0 thành công, -1 lỗi. Lỗi thêm: EINVAL nếu flags không hợp lệ
Cách dùng pipe2() đúng chuẩn cho embedded/production
#include <errno.h>   /* errno, EAGAIN, EWOULDBLOCK */
#include <fcntl.h>   /* O_CLOEXEC, O_NONBLOCK */
#include <unistd.h>  /* pipe2(), read() */

int fds[2];

/* pipe2 = pipe + O_CLOEXEC + O_NONBLOCK trong 1 syscall,
   tránh race condition giữa pipe() và fcntl() */
if (pipe2(fds, O_CLOEXEC | O_NONBLOCK) == -1) {
    perror("pipe2");
    return -1;
}

/* Đọc non-blocking */
char buf[256];
ssize_t n = read(fds[0], buf, sizeof(buf));
if (n == -1) {
    if (errno == EAGAIN || errno == EWOULDBLOCK) {
        /* Bình thường: không có dữ liệu lúc này */
    } else {
        perror("read");  /* Lỗi thực sự */
    }
}
write() và read() trong ngữ cảnh pipe
#include <unistd.h> ssize_t write(int fd, const void *buf, size_t count); ssize_t read (int fd, void *buf, size_t count);
fd
Với pipe: fds[1] cho write, fds[0] cho read. Dùng sai (ghi vào [0] hoặc đọc từ [1]) → EBADF.
buf
Con trỏ đến vùng nhớ chứa dữ liệu (write) hoặc nơi nhận dữ liệu (read). Phải valid và đủ kích thước.
count
Số bytes muốn ghi/đọc. Kết quả thực tế có thể ít hơn — luôn kiểm tra giá trị trả về.
Trả về: số bytes thực sự đọc/ghi (≥0), hoặc -1 lỗi. Riêng read: 0 = EOF
Partial write: write() có thể ghi ít hơn count bytes, đặc biệt với pipe lớn hoặc signal ngắt. Phải kiểm tra giá trị trả về và ghi lại phần còn thiếu — đây là lỗi rất phổ biến trong code production.
write_all() — helper function viết đủ bytes
/* Đảm bảo ghi đủ n bytes, xử lý partial write và EINTR */
ssize_t write_all(int fd, const void *buf, size_t n) {
    const char *p = buf;
    size_t remaining = n;
    while (remaining > 0) {
        ssize_t w = write(fd, p, remaining);
        if (w == -1) {
            if (errno == EINTR) continue;  /* Signal ngắt, thử lại */
            return -1;                     /* Lỗi thực sự */
        }
        p         += w;
        remaining -= w;
    }
    return (ssize_t)n;
}
FIFO — Named Pipe, có tên trên filesystem
Giống anonymous pipe nhưng có path, dùng được giữa các process không liên quan.
Điểm khác biệt so với anonymous pipe
Anonymous pipe
— Không có tên trên filesystem
— Dùng giữa parent và child (kế thừa fd qua fork), hoặc giữa các thread trong cùng process (chia sẻ fd table)
— Mất khi tất cả fd đã đóng (process chết → fd tự đóng)
— Tạo bằng pipe()
FIFO (Named pipe)
— Có inode trên filesystem (thường ở /tmp)
— Bất kỳ process nào biết path đều dùng được, kể cả không liên quan nhau
— File FIFO tồn tại trên filesystem cho đến khi bị unlink() (nhưng kernel buffer mất khi tất cả fd đóng)
— Tạo bằng mkfifo()
mkfifo() — tạo FIFO
#include <sys/types.h> #include <sys/stat.h> int mkfifo(const char *pathname, mode_t mode);
pathname
Đường dẫn tạo FIFO trên filesystem. Ví dụ: "/tmp/my_fifo" hoặc "/var/run/myapp/cmd". Thư mục cha phải tồn tại.
mode
Permission bits, giống chmod. Ví dụ: 0666 = đọc/ghi cho tất cả. Giá trị thực tế bị mask bởi umask của process. Trong embedded thường dùng 0600 (chỉ owner đọc/ghi) hoặc 0660 (owner + group).
Trả về: 0 thành công, -1 lỗi. Lỗi quan trọng: EEXIST = path đã tồn tại, ENOENT = thư mục cha không tồn tại
Hành vi quan trọng: open() FIFO block cho đến khi cả 2 đầu sẵn sàng
Đây là điểm khác biệt lớn nhất so với anonymous pipe. Khi mở FIFO:

— open(path, O_RDONLY): block cho đến khi có process mở O_WRONLY
— open(path, O_WRONLY): block cho đến khi có process mở O_RDONLY

Lý do: kernel cần cả 2 đầu (reader + writer) sẵn sàng thì pipe mới có ý nghĩa hoạt động.
Khi dùng O_NONBLOCK với open() trên FIFO:

— open(path, O_RDONLY | O_NONBLOCK): thành công ngay lập tức, trả về fd hợp lệ dù chưa có writer. Khi read() sau đó mà chưa có writer, trả về -1 với errno = EAGAIN.
— open(path, O_WRONLY | O_NONBLOCK): nếu chưa có reader, open() thất bại ngay với errno = ENXIO. Đây là hành vi bất đối xứng so với reader — rất dễ gây nhầm lẫn.

Lý do bất đối xứng: Reader mở trước là hợp lý (chờ writer sẵn sàng), nhưng writer mở khi không có reader thì dữ liệu ghi sẽ không ai đọc → kernel từ chối luôn.
fifo_server.c — bên nhận lệnh (chạy trước)
#include <sys/stat.h>  /* mkfifo() */
#include <fcntl.h>     /* open(), O_RDONLY */
#include <unistd.h>    /* read(), close(), unlink() */
#include <stdio.h>
#include <errno.h>     /* errno, EEXIST */

#define FIFO_PATH "/tmp/cmd_fifo"

int main(void) {
    /* Tạo FIFO nếu chưa có */
    if (mkfifo(FIFO_PATH, 0660) == -1) {
        if (errno != EEXIST) {   /* EEXIST = đã tồn tại, OK */
            perror("mkfifo"); return 1;
        }
    }
    printf("Dang cho lenh... (FIFO: %s)\n", FIFO_PATH);

    /* open() sẽ BLOCK ở đây cho đến khi client mở để ghi */
    int fd = open(FIFO_PATH, O_RDONLY);
    if (fd == -1) { perror("open"); return 1; }

    char buf[256];
    ssize_t n;
    while ((n = read(fd, buf, sizeof(buf)-1)) > 0) {
        buf[n] = '\0';
        printf("Nhan lenh: [%s]\n", buf);
    }
    close(fd);

    /* Dọn dẹp — FIFO vẫn tồn tại trên disk nếu không unlink */
    unlink(FIFO_PATH);
    return 0;
}
fifo_client.c — bên gửi lệnh (chạy sau)
#include <fcntl.h>
#include <unistd.h>
#include <stdio.h>
#include <string.h>

#define FIFO_PATH "/tmp/cmd_fifo"

int main(void) {
    /* open() block cho đến khi server mở O_RDONLY */
    int fd = open(FIFO_PATH, O_WRONLY);
    if (fd == -1) { perror("open"); return 1; }

    const char *cmd = "START_SENSOR";
    write(fd, cmd, strlen(cmd));
    printf("Da gui: %s\n", cmd);

    close(fd);  /* Đóng → server thấy EOF */
    return 0;
}

Pipe & FIFO trong Embedded Linux
Những điểm khác biệt, ưu tiên, và giới hạn khi dùng trên thiết bị nhúng.
Những ràng buộc đặc thù của embedded Linux
RAM giới hạn: Buffer mặc định 64 KB/pipe. Trên thiết bị 64 MB RAM, tạo nhiều pipe có thể chiếm đáng kể tài nguyên. Kiểm tra: cat /proc/sys/fs/pipe-max-size
Filesystem read-only: FIFO cần ghi vào filesystem. Trên hệ thống có rootfs read-only (SquashFS + overlayfs), phải tạo FIFO ở /tmp (tmpfs) hoặc /var/run (ramfs), không phải /etc hay /usr.
Ít fd hơn: ulimit -n trên embedded thường là 256–1024. Pipe tốn 2 fd. Luôn close() fd không cần thiết ngay sau fork().
Không có /proc/sys/fs/pipe-max-size trên kernel cũ: Buildroot với kernel < 3.5 có thể không có F_GETPIPE_SZ / F_SETPIPE_SZ. Kiểm tra bằng uname -r trước khi dùng.
SIGPIPE có thể kill daemon: Nếu daemon ghi vào pipe mà reader đã chết, SIGPIPE kill daemon. Luôn ignore hoặc handle SIGPIPE trong daemon code.
Pattern thực tế: pipe giữa main task và sensor task
embedded_pipe.c — pattern dùng pipe + select() để không block main loop (pseudo-code)
/* Đây là pseudo-code minh họa pattern, không phải chương trình hoàn chỉnh.
   sensor_thread() và main_loop() là 2 hàm riêng biệt chạy ở 2 thread. */

#include <unistd.h>
#include <fcntl.h>
#include <signal.h>
#include <sys/select.h>
#include <errno.h>

static int fds[2];  /* Pipe fd dùng chung giữa 2 thread */

/* ── Khởi tạo trong main() ── */
void init(void) {
    /* Ignore SIGPIPE — quan trọng cho daemon.
       Nên dùng sigaction() thay signal() cho portable code,
       nhưng signal(SIG_IGN) là ngoại lệ an toàn. */
    signal(SIGPIPE, SIG_IGN);

    /* O_CLOEXEC: tự đóng khi exec — bắt buộc trên embedded
       để tránh rò fd khi spawn subprocess (busybox, etc.) */
    pipe2(fds, O_CLOEXEC | O_NONBLOCK);
}

/* ── Sensor thread ghi kết quả ── */
void sensor_thread(void) {
    float temperature = read_sensor();  /* Hàm giả định đọc cảm biến */
    /* sizeof(float) = 4 bytes < PIPE_BUF = 4096 → atomic, an toàn.
       Nhiều thread ghi cùng lúc cũng không bị xen lẫn. */
    write(fds[1], &temperature, sizeof(float));
}

/* ── Main loop đọc không blocking bằng select() ── */
void main_loop(void) {
    fd_set rfds;
    struct timeval tv = { .tv_sec = 0, .tv_usec = 10000 }; /* 10ms timeout */

    FD_ZERO(&rfds);
    FD_SET(fds[0], &rfds);

    int ret = select(fds[0] + 1, &rfds, NULL, NULL, &tv);
    if (ret > 0 && FD_ISSET(fds[0], &rfds)) {
        float temp;
        if (read(fds[0], &temp, sizeof(float)) == sizeof(float)) {
            handle_temperature(temp);  /* Hàm giả định xử lý nhiệt độ */
        }
    }
    /* ret == 0 → timeout, không có data, tiếp tục loop bình thường */
}
Khi nào dùng Pipe vs FIFO trong embedded
Dùng Pipe khi
— Giao tiếp giữa parent và child process (sau fork)
— Giao tiếp giữa các thread trong cùng process
— Cần self-pipe trick để wake up select/epoll từ signal handler
— Không muốn để lại vết trên filesystem
Dùng FIFO khi
— Hai daemon độc lập cần giao tiếp (không có quan hệ cha-con)
— Cần giao diện đơn giản: echo "cmd" > /tmp/fifo
— Script shell cần giao tiếp với C daemon
— Debugging: có thể cat /tmp/fifo để xem data
Self-pipe trick — đánh thức main loop từ signal handler
Bài toán thực tế
Hãy tưởng tượng bạn viết một daemon (chương trình chạy nền) trên embedded Linux. Daemon này có một main loop — vòng lặp chính chạy liên tục, đợi dữ liệu từ sensor, socket, pipe... bằng epoll_wait() hoặc select().

Bây giờ bạn muốn: khi hệ thống gửi lệnh tắt daemon (signal SIGTERM) hoặc lệnh reload cấu hình (signal SIGUSR1), daemon phải phản ứng ngay lập tức — nhưng epoll_wait() đang ngủ chờ dữ liệu. Làm sao đánh thức nó?
Signal là gì?
Signal là cơ chế kernel dùng để gửi thông báo bất đồng bộ (asynchronous notification) đến một process. "Bất đồng bộ" nghĩa là signal có thể đến bất kỳ lúc nào — giữa lúc process đang làm gì đó, thậm chí giữa chừng một dòng code.

Một số signal phổ biến:
— SIGTERM (15): "Hãy tắt đi" — lệnh tắt lịch sự, cho process cơ hội dọn dẹp
— SIGKILL (9): "Tắt ngay" — không thể bắt hay bỏ qua, kernel kill ngay lập tức
— SIGUSR1 (10): Signal do người dùng tự định nghĩa mục đích (thường dùng để reload config)
— SIGPIPE (13): "Pipe đã hỏng" — ghi vào pipe mà reader đã đóng

Gửi signal: kill -SIGTERM 1234 (gửi SIGTERM đến process có PID 1234).
Signal handler là gì? Và tại sao nó bị hạn chế?
Signal handler là hàm bạn đăng ký để chạy khi process nhận được signal cụ thể. Kernel sẽ tạm dừng code đang chạy, nhảy vào signal handler, chạy xong rồi quay lại chỗ cũ.

Vấn đề nguy hiểm: Vì signal đến bất kỳ lúc nào, handler có thể chạy giữa chừng một hàm khác. Ví dụ:

— malloc() đang cập nhật cấu trúc heap → signal đến → handler gọi malloc() lần nữa → corrupt heap → crash
— printf() đang ghi buffer → signal đến → handler gọi printf() → output lẫn lộn hoặc deadlock (vì printf dùng mutex nội bộ)

Vì lý do này, POSIX quy định: trong signal handler, chỉ được gọi các hàm "async-signal-safe" — là những hàm đảm bảo an toàn khi bị gọi lại giữa chừng. Danh sách này rất ngắn, chủ yếu là các syscall đơn giản: read(), write(), close(), _exit()...

Tóm lại: Trong signal handler, bạn KHÔNG được gọi printf(), malloc(), free(), lock mutex, hay bất kỳ hàm phức tạp nào. Nhưng write() thì AN TOÀN.
💡 Ý tưởng Self-pipe trick
Vì signal handler không được làm gì phức tạp, nhưng write() thì an toàn — ta lợi dụng điều này:

Bước 1: Tạo một pipe "nội bộ" (self-pipe) — không để giao tiếp giữa 2 process, mà chỉ để đánh thức main loop.
Bước 2: Thêm đầu đọc pipe[0] vào danh sách mà epoll_wait()/select() đang theo dõi.
Bước 3: Trong signal handler, chỉ làm 1 việc: write(pipe[1], &byte, 1) — ghi 1 byte vào pipe.
Bước 4: epoll_wait() đang ngủ → thấy pipe có dữ liệu → thức dậy → main loop đọc byte ra, xử lý signal một cách an toàn (vì bây giờ đang ở main loop, không phải trong signal handler — được gọi bất kỳ hàm nào).
Tại sao gọi là "self-pipe"? Vì pipe này không dùng để giao tiếp giữa 2 process — process tự tạo pipe, tự ghi (trong signal handler), tự đọc (trong main loop). Nó như tự nói chuyện với chính mình qua pipe.
self_pipe_trick.c — ví dụ hoàn chỉnh, chạy được
#include <unistd.h>
#include <fcntl.h>
#include <signal.h>
#include <sys/epoll.h>
#include <stdio.h>

/* ── Pipe "nội bộ" — global để signal handler truy cập được ── */
static int wakeup_pipe[2];

/* ── Signal handler ──
   Hàm này bị gọi BẤT KỲ LÚC NÀO khi process nhận signal.
   Quy tắc: CHỈ được gọi async-signal-safe functions.
   write() là async-signal-safe → an toàn. */
void sig_handler(int sig) {
    char byte = (char)sig;  /* Gửi số hiệu signal qua pipe */
    write(wakeup_pipe[1], &byte, 1);
    /* Không kiểm tra return value — cố ý.
       Nếu pipe đầy (gần như không thể xảy ra), write() trả -1
       nhưng không crash. Trong signal handler, càng ít code càng tốt. */
}

int main(void) {
    /* ── Bước 1: Tạo pipe non-blocking ── */
    pipe2(wakeup_pipe, O_CLOEXEC | O_NONBLOCK);
    /* O_NONBLOCK: để write() trong signal handler không bao giờ block.
       O_CLOEXEC: tự đóng fd khi gọi exec(), tránh rò rỉ fd. */

    /* ── Bước 2: Đăng ký signal handler ──
       sigaction() là cách chuẩn, tốt hơn signal().
       Khi process nhận SIGTERM hoặc SIGUSR1, kernel sẽ gọi sig_handler(). */
    struct sigaction sa = { .sa_handler = sig_handler };
    sigemptyset(&sa.sa_mask);  /* Không block thêm signal nào khi handler chạy */
    sa.sa_flags = 0;
    sigaction(SIGTERM, &sa, NULL);  /* Đăng ký cho SIGTERM */
    sigaction(SIGUSR1, &sa, NULL);  /* Đăng ký cho SIGUSR1 */

    /* ── Bước 3: Tạo epoll và thêm pipe vào ──
       epoll là cơ chế Linux dùng để "chờ" nhiều fd cùng lúc.
       epoll_wait() sẽ ngủ cho đến khi ÍT NHẤT 1 fd có dữ liệu.
       Ta thêm wakeup_pipe[0] vào epoll → khi signal handler ghi 1 byte
       vào wakeup_pipe[1], epoll_wait() sẽ thức dậy. */
    int epfd = epoll_create1(EPOLL_CLOEXEC);
    struct epoll_event ev = {
        .events = EPOLLIN,              /* Theo dõi sự kiện "có data để đọc" */
        .data.fd = wakeup_pipe[0]       /* fd cần theo dõi */
    };
    epoll_ctl(epfd, EPOLL_CTL_ADD, wakeup_pipe[0], &ev);

    printf("Daemon dang chay (PID=%d). Thu gui signal:\n", getpid());
    printf("  kill -SIGUSR1 %d   (reload)\n", getpid());
    printf("  kill -SIGTERM %d   (shutdown)\n\n", getpid());

    /* ── Bước 4: Main loop ── */
    struct epoll_event events[4];
    for (;;) {
        /* epoll_wait() ngủ ở đây cho đến khi có event.
           -1 = chờ vô hạn (không timeout).
           Khi signal đến → handler ghi byte vào pipe → epoll thức dậy. */
        int n = epoll_wait(epfd, events, 4, -1);

        for (int i = 0; i < n; i++) {
            if (events[i].data.fd == wakeup_pipe[0]) {
                /* Pipe có data → chắc chắn signal handler vừa ghi byte */
                char sig_byte;
                read(wakeup_pipe[0], &sig_byte, 1);

                /* BÂY GIỜ ta đang ở main loop, KHÔNG phải signal handler.
                   → Thoải mái gọi printf, malloc, free, log... */
                if (sig_byte == SIGTERM) {
                    printf("Nhan SIGTERM → shutdown...\n");
                    goto cleanup;
                } else if (sig_byte == SIGUSR1) {
                    printf("Nhan SIGUSR1 → reload config...\n");
                    /* Gọi reload_config() hay bất kỳ hàm phức tạp nào ở đây */
                }
            }
            /* Nếu epoll theo dõi thêm fd khác (socket, sensor pipe...),
               xử lý chúng ở đây tương tự. */
        }
    }
cleanup:
    close(epfd);
    close(wakeup_pipe[0]);
    close(wakeup_pipe[1]);
    printf("Daemon da thoat.\n");
    return 0;
}
Lưu ý: Trên Linux ≥ 2.6.22, có thể dùng signalfd() thay thế self-pipe trick — cũng trả về fd để dùng với epoll, nhưng API gọn hơn. Tuy nhiên self-pipe trick là portable hơn (hoạt động trên mọi POSIX system, kể cả macOS, FreeBSD).
Bước tiếp theo: Sau khi nắm vững Pipe & FIFO, IPC tiếp theo nên học là Unix Domain Socket — cho phép bidirectional, nhiều client, và truyền file descriptor. Đặc biệt hữu ích cho kiến trúc daemon-client trong embedded Linux.
