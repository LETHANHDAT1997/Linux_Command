User Space
Unprivileged execution ring (Ring 3). Programs here cannot directly access hardware or kernel memory. All hardware/OS interaction must go through the System Call interface.
Everything you interact with — your shell, browser, Python scripts — runs here. The kernel enforces strict isolation: a crash in user space cannot bring down the kernel. Memory is virtual and protected per-process.

glibc — GNU C Library

glibc is the bridge between user programs and the kernel. When your C program calls printf() or malloc(), it's calling glibc. glibc translates these into the correct system calls, handles buffering, locale, threading (pthreads), and math. Nearly every Linux binary links against it.

Example — printf() path through glibc 
/* Your program calls: */
printf("Hello, world!\n");

/* glibc internally does (simplified): */
write(STDOUT_FILENO, buf, len);   // calls write(2) syscall
// → syscall instruction → kernel sys_write()
Inspect glibc linkage 
ldd /bin/ls            # show shared libs, glibc is always present
strace ls /tmp        # trace every syscall ls makes via glibc
Shell — bash / zsh / fish

A shell is a command interpreter running as a regular user-space process. It reads your commands, forks child processes via fork()+execve(), sets up pipes using pipe(), and handles job control via signals. The shell itself uses glibc and issues syscalls just like any program.

What happens when you type: ls | grep foo 
# Bash does approximately:
pipe(fds);          // syscall: create pipe
fork();             // syscall: clone process for "ls"
  dup2(fds[1], 1);  // redirect ls stdout → pipe write end
  execve("/bin/ls", ...);
fork();             // clone again for "grep"
  dup2(fds[0], 0);  // redirect grep stdin ← pipe read end
  execve("/bin/grep", ...);
Applications — Python, nginx, browsers…

High-level applications sit furthest from the hardware. A Python script calls Python's runtime, which calls glibc, which issues syscalls. nginx uses epoll syscalls for async I/O. Every application is constrained to its virtual address space and cannot access another process's memory directly.

Python open() call chain 
# Python source:
f = open("/etc/hostname", "r")

# CPython → glibc fopen() → sys_openat() syscall
# Kernel checks DAC permissions, returns fd integer
strace -e openat python3 -c "open('/etc/hostname')"
Programs — Compiled Binaries (ELF)

Linux executables use the ELF (Executable and Linkable Format). The kernel's execve() handler reads the ELF header, maps segments into virtual memory, sets up the stack, and jumps to the entry point. Dynamically linked programs also invoke the dynamic linker (ld.so) to resolve shared library symbols.

Inspect an ELF binary 
file /bin/bash          # "ELF 64-bit LSB pie executable…"
readelf -h /bin/bash    # ELF header: entry point, sections
readelf -l /bin/bash    # Program headers: LOAD, DYNAMIC…
objdump -d /bin/ls      # Disassemble to see actual instructions

The only legal crossing point between User Space and Kernel Space

Kernel Space
Privileged execution (Ring 0). Full hardware access, no memory protection from itself. A bug here can crash or corrupt the entire system. The kernel is not a process — it runs on behalf of processes.
⚙️ Process Management ▼

The kernel tracks every process in a task_struct — a C struct containing PID, state (running/sleeping/zombie), open file descriptors, signal handlers, scheduling priority, memory map, and more. The Completely Fair Scheduler (CFS) decides which process runs next on each CPU core using a red-black tree ordered by virtual runtime.

Process states: TASK_RUNNING → runnable or running, TASK_INTERRUPTIBLE → sleeping (wakes on signal), TASK_UNINTERRUPTIBLE → sleeping in kernel (disk I/O wait), TASK_ZOMBIE → exited, awaiting parent's wait().

Inspect processes 
ps aux                   # snapshot: PID, state, CPU%, MEM%
top / htop               # live view; press 'f' for fields
cat /proc/1/status        # init's task_struct info in text
cat /proc/1/wchan         # kernel fn where proc is sleeping
ls /proc/$$/fd/           # open file descriptors of current shell

🧠 Memory Management ▼

Each process has its own virtual address space (0 to 2⁴⁸ on x86-64). The MMU hardware translates virtual addresses to physical RAM using multi-level page tables. The kernel manages physical pages via the buddy allocator (for large blocks) and slab allocator (for kernel objects). When physical memory is scarce, the kernel swaps pages to disk using an LRU-based page replacement policy.

Copy-on-Write: after fork(), parent and child share the same physical pages marked read-only — a page is only copied when one side writes to it. This makes fork() extremely cheap.

Inspect memory 
cat /proc/meminfo         # MemTotal, MemFree, Cached, SwapUsed…
cat /proc/$$/maps         # virtual address layout of this shell
cat /proc/$$/smaps        # per-VMA stats: RSS, PSS, shared…
free -h                   # human-readable memory summary
vmstat 1                  # paging activity per second

📁 Virtual File System (VFS) ▼

VFS is an abstraction layer that gives every file system (ext4, xfs, tmpfs, procfs, sysfs…) a uniform interface. It defines four key objects: superblock (mounted filesystem metadata), inode (file metadata: permissions, timestamps, size), dentry (directory entry / path component cache), and file (open file state for a process).

open("/etc/passwd") → VFS looks up dentries → reaches inode → dispatches to ext4's read() implementation → returns data. The calling code never knows which filesystem is beneath.

Explore the VFS 
mount                     # show mounted filesystems
ls /proc/$$/fd/           # file descriptors → VFS file objects
stat /etc/passwd          # inode number, link count, times
df -Th                    # fstype, usage per superblock
cat /proc/filesystems     # all registered fs types in kernel
ls /proc/                 # procfs: virtual files backed by kernel
inodedentry cache ext4 / xfs / btrfsprocfs sysfstmpfs
🔌 Device Drivers ▼

Drivers are kernel code that know how to talk to specific hardware. Linux categorizes devices as character (stream of bytes: terminals, serial ports), block (random-access chunks: disks, SSDs), or network (packet-based: eth0, wlan0). Every device gets a major (driver type) and minor (instance) number. User-space accesses devices through /dev device files.

The udev daemon watches for hardware events via netlink sockets and automatically creates/removes /dev nodes and loads modules with modprobe.

Inspect devices 
ls -l /dev/sda /dev/tty0   # block (b) and char (c) devices
cat /proc/devices          # registered major numbers
lsblk                      # block device tree
lspci -k                   # PCI devices + which driver loaded
udevadm monitor            # watch udev events in real-time
dd if=/dev/urandom bs=1 count=8 | xxd  # read from char device

🌐 Networking Stack ▼

Linux implements a full BSD-compatible socket API on top of a layered networking stack. Sockets are the user-space handle; beneath them sit the transport layer (TCP/UDP via net/ipv4/tcp.c), IP layer (routing, NAT), and the NIC driver. sk_buff (socket buffer) is the kernel struct that carries a packet through every layer.

netfilter hooks into the packet path for iptables/nftables firewall rules and NAT. eBPF programs can be attached at various hooks for ultra-fast packet processing (XDP) without leaving the kernel.

Inspect the networking stack 
ip addr show              # interfaces and addresses
ss -tunaop               # socket table (replaces netstat)
cat /proc/net/tcp        # raw kernel TCP socket table
ip route show            # kernel routing table
iptables -L -n -v        # netfilter firewall rules
tcpdump -i eth0 -nn      # capture packets at driver level

🔗 IPC — Inter-Process Communication ▼

Processes are isolated by design, so the kernel provides explicit channels for communication. IPC mechanisms differ in performance, persistence, and synchronization needs:

Pipes — unidirectional byte stream, kernel buffer (~64 KB), only between related processes. Named pipes (FIFOs) — same but via filesystem path. Unix sockets — full-duplex, datagram or stream, same host only. Shared memory — fastest; processes map the same physical pages. Message queues — typed, discrete messages, kernel-buffered. Semaphores — synchronization primitives for shared resources. Signals — asynchronous notifications (lightweight, no data payload).

Explore IPC 
ipcs                    # list SysV IPC objects (shm/msg/sem)
ls /proc/sysvipc/       # kernel IPC tables
ls /dev/shm/            # POSIX shared memory objects
# Test a pipe:
echo "hello" | cat      # shell creates pipe, fork, connect
# Unix socket in use by dbus:
ss -x                   # list Unix domain sockets
pipe/FIFOUnix socket shared memorysemaphore signalmessage queue
🛡️ Security (LSM / DAC / MAC) ▼

DAC (Discretionary Access Control) — the classic Unix model: every file has owner UID/GID and rwx permission bits. The kernel checks these on every open/exec.

Linux Security Module (LSM) — a hook framework letting security modules (SELinux, AppArmor, Smack) attach mandatory access control checks to every sensitive kernel operation. SELinux labels every object (file, socket, process) with a security context and enforces a policy database.

Capabilities — traditional root/non-root split replaced by ~40 granular capabilities (e.g. CAP_NET_BIND_SERVICE to bind port <1024, CAP_SYS_PTRACE to debug processes).

Namespaces + cgroups — foundation of containers; namespaces isolate view of pid/net/mount/user, cgroups limit CPU/memory/I/O.

Inspect security 
ls -l /etc/shadow          # DAC: only root can read
getcap /bin/ping           # see capability on a file
cat /proc/$$/status | grep Cap  # hex caps of shell
getenforce                 # SELinux mode: Enforcing/Permissive
aa-status                  # AppArmor profiles loaded
unshare --user --pid --fork bash  # enter new namespaces

⚡ Interrupt Handling ▼

Hardware devices signal the CPU asynchronously via IRQ lines. When a NIC receives a packet or a keyboard key is pressed, the CPU pauses current work, saves registers, and jumps to the registered Interrupt Service Routine (ISR) in the kernel.

To avoid blocking other interrupts too long, Linux splits interrupt handling into a fast top half (runs in IRQ context: saves data, acknowledges device) and a deferred bottom half (runs later as a softirq, tasklet, or workqueue: does heavy processing). Softirqs handle network Rx/Tx and timer ticks. The timer interrupt at HZ frequency (typically 250 Hz) drives the scheduler tick.

Inspect interrupts 
cat /proc/interrupts       # IRQ # → count per CPU → name
watch -n1 cat /proc/interrupts  # watch counts increment live
cat /proc/softirqs        # softirq counts: NET_RX, TIMER…
# Time a tight loop to see timer interrupt cost:
cyclictest -l 10000 -m -n  # real-time latency tester

🧩 Module System (LKM) ▼

Loadable Kernel Modules (LKMs) allow drivers and subsystems to be compiled separately and inserted into the running kernel without a reboot. Modules are ELF .ko files. insmod loads the object, the kernel resolves symbols against the live kernel symbol table, and calls the module's init_module(). rmmod calls cleanup_module() and unlinks it.

Modules run in Ring 0 with full kernel privileges — a buggy module can crash the system. modprobe is smarter than insmod: it reads /lib/modules/$(uname -r)/modules.dep to auto-load dependencies. The kernel can auto-load modules on demand using module aliases (e.g. plugging a USB device triggers udev → modprobe with the USB vendor:product alias).

Work with modules 
lsmod                     # loaded modules + size + use count
modinfo ext4              # description, params, depends, alias
modprobe loop             # load loop block driver + deps
modprobe -r loop          # unload loop (if not in use)
ls /lib/modules/$(uname -r)/  # all available .ko files
dmesg | grep -i module    # kernel log: module load/unload events
